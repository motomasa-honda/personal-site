## TL;DR

生成AIが書いたコードを実行するとき、環境変数を極限まで絞った「許可リスト」方式のサンドボックスを
用意していた。だがこれまで検証していたのは、事前学習済みモデルの読み込みや行列演算といった
「読むだけ」の処理ばかりだった。本当に重要なのは、実際に勾配を計算して重みを更新する学習
ループが同じ制限の中で回るかどうかだ。実機で確かめたところ、途中で一度「サンドボックスが
壊した」と誤解しそうになる失敗に出会った。原因を切り分けた話を書く。

---

## サンドボックスは「読むだけ」しか試していなかった

生成AIにコードを書かせて、それをそのまま実行する仕組みを持っている。書かれたコードが
うっかり(あるいは悪意を持って)外部に何かを送信したり、環境変数経由で秘密情報を持ち出したり
しないように、実行時に渡す環境変数を数個(パス関連の最低限)だけに絞る、という許可リスト方式の
サンドボックスを用意していた。

この許可リストが実際のマシンで本当に機能するかは、以前に一度実機で検証済みだった。ただし
そのとき試したのは、表形式データの学習、事前学習済み画像モデルのダウンロード、GPU上での
行列演算——どれも「モデルを読み込んで使う」側の処理だった。

本当に確かめたかったのはそこではない。生成AIに書かせるコードの本命は「学習ループ」だ。
勾配を計算し、重みを更新し、結果を保存する。読むだけの処理と違って、書き込み先のキャッシュ
ディレクトリ、GPU上でのオプティマイザ状態、チェックポイントの保存——制限された環境変数の
中でも本当に全部通るのか、まだ誰も確かめていなかった。

## 実際に学習ループを回してみる

小さな分類モデルに対して、フルの環境変数がある状態(比較用の基準)と、パス関連の4つの
変数だけに絞った状態の両方で、同じ学習コードを実行した。

```python
model = AutoModelForSequenceClassification.from_pretrained(model_name, num_labels=2)
...
out = model(**batch, labels=labels)
out.loss.backward()
optimizer.step()
```

これはあっさり両方の環境で成功した。次に、まだキャッシュに存在しない新しいモデルを
ダウンロードして使う処理も試した。ここで最初のつまずきが起きた。

```
ValueError: Couldn't instantiate the backend tokenizer from one of:
(1) a `tokenizers` library serialization file,
(2) a slow tokenizer instance to convert or
(3) an equivalent slow tokenizer class to instantiate and convert.
You need to have sentencepiece or tiktoken installed to convert a slow tokenizer to a fast one.
```

一瞬「制限した環境変数のせいでトークナイザのライブラリが見つからなくなったのでは」と
疑いかけた。だが慌てて結論を出す前に、**同じスクリプトを制限なしのフル環境でも走らせて
比較する**、という基本に立ち返った。結果は同じエラー。原因は環境変数の制限とは無関係で、
単にそのマシンに`sentencepiece`というパッケージ自体が入っていなかっただけだった。ダウンロード
自体はきちんと完了していて(キャッシュにファイルが残っていることも確認した)、失敗していたのは
その後のトークナイザ変換処理だけだった。

## 本命: LoRAアダプタの学習まで通すと

次に、実際に不足していたライブラリを入れて、LoRAアダプタの学習まで確かめた。全パラメータの
うち1%未満だけを学習対象にする、という本来の使い方通りの設定にした。

```python
lora_config = LoraConfig(task_type="SEQ_CLS", r=4, lora_alpha=8, target_modules=["q_lin", "v_lin"])
model = get_peft_model(base_model, lora_config)
# trainable_params=665858 total_params=67620868 (0.98%)
```

2エポック分の学習ループを回し、アダプタを保存するところまで、フル環境と制限環境の両方で
実行した。結果は完全に一致——学習後の損失の値も、保存されたアダプタファイルの構成も同じ
だった。GPUの利用、勾配計算、オプティマイザの状態更新、チェックポイントの書き込み、すべてが
数個の環境変数だけで問題なく動いた。

## 今日の教訓

一つの失敗の原因を、目の前で疑わしく見える要素(今回なら「環境変数を絞ったこと」)にすぐ
結びつけるのは早計だった。**同じ条件をフル環境でも再現して比較する**という一手間が、
「サンドボックスのせい」という誤った結論を防いだ。もし比較せずに「制限がきつすぎたか」と
判断していたら、実際には無関係なパッケージ不足の問題を追いかけて許可リストを不必要に
緩めていたかもしれない。

もう一つ。サンドボックスの検証は「代表的な使い方を一通り試した」で終わらせず、
**そのサンドボックスが本来守るべき、一番負荷の高い本命の処理**(今回なら学習ループそのもの)
まで実際に通してみて初めて、安心して「機能する」と言える。読むだけの処理が通ることは、
書き込みを伴う処理が通ることの証明にはならない。
