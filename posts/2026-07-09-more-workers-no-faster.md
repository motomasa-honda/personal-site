## TL;DR

自分で書いた仕様書に「Worker Poolのサイズを増やせば、そのぶんタスクが並列に走る」と
はっきり書いていた。設定ファイルにも「この2行を書き換えるだけでスケールする」という
コメントまで残していた。だが実際にコードを読み返すと、Workerを何体増やしても実行は
一切並列化されていなかった。原因は、実行ループ自体が「1回呼ぶと1つのタスクが終わるまで
ブロックする」という作りのままだったこと。直した話と、直す過程で3年前の自分と同じ
バグをもう一箇所で踏んでいたのを見つけた話を書く。

---

## 仕様書は正しかった。実装が追いついていなかった

このシステムには「Worker Poolのサイズが1台の個人PCから百台規模のデータセンターまで、
唯一スケールする変数である」と定めた仕様書がある。Worker Poolを1にすれば厳密に順次実行、
Nにすれば最大N個のタスクが同時に走る——そう書いてあった。設定ファイルのコメントにも
「トリガー①が来たらこの数値を3から20に上げるだけでいい」と自分で残していた。

読み返してみると、実行の中核にあるループはこう書かれていた。

```python
def run_once(self, handler):
    result_task = self._scheduler.assign(handler)  # ここでブロックする
    ...
    return result_task

def run_until_idle(self, handler):
    while True:
        task = self.run_once(handler)  # 1つ終わるまで次を呼ばない
        if task is None:
            break
```

`assign()` は「空いているWorkerを1体探して、そのWorkerにタスクを1つ実行させて、
終わるまで待つ」という処理を1回の呼び出しでやっていた。呼び出し元のループは、
それが返ってくるまで次のタスクに進まない。つまりWorkerを何体プールに入れていようと、
実際に動いているのは常に1体だけだった。

`Scheduler` 自身のdocstringには「Concurrency Tier 1では厳密に順次実行される」と
既に書いてあった。上位のティアを見据えていた形跡はあったが、実装されたことは
一度もなかった。設定ファイルの「この2行を書き換えるだけ」というコメントは、
実際には何の効果も持たない約束を書いていたことになる。

## 直し方: 既存の契約を1文字も壊さない

一番気を遣ったのは、既存の呼び出し側とテストを1つも壊さないことだった。
`run_once`/`run_until_idle`(順次実行版)はそのまま残し、新しく `run_concurrently`
という並列版を追加する形にした。

内部では、タスクの「予約」と「実行」を分離した。

- `Scheduler.claim()` — ロックの中で「空いているWorkerを1体見つけて、実行可能な
  タスクを1つ確保して、Workerを予約済み(BUSY)にする」ところまでを、実際にコードを
  走らせずに一瞬で終える
- `Kernel.run_concurrently()` — 空いているWorkerの数だけ `claim()` を繰り返して
  タスクを確保し、それぞれをスレッドプールに投げて、どれか1つ終わるたびに次を確保する

こうすると、既存の `Scheduler.assign()`(順次版)は「`claim()` して、その場で
実行して、終わるまで待つ」という組み合わせとして書き直せる。外から見た挙動は
1ミリも変わらない。実際、既存のテストは1行も書き換えずに全部通った。

## 予約の途中に隠れていた競合状態

このリファクタの過程で、既存コードにもう1つ問題を見つけた。「次に実行すべき
タスクを探す」処理と、「そのタスクをRUNNING状態にして保存する」処理が、
別々の呼び出しに分かれていた。

```python
task = self._queue.next_pending()   # ここでロックが外れる
task.transition_to(TaskState.RUNNING)
self._queue.save(task)              # 別のロックで保存
```

これは前に書いた「採番と保存が別の操作だったせいで起きる競合」バグと
全く同じ形をしている。2つのスレッドが同時に呼ぶと、両方とも同じタスクを
「未着手」だと読んでしまい、両方が実行を始めてしまう。今回並列実行を
実装するまでは1スレッドしか呼んでいなかったので表面化していなかったが、
まさにこれから複数スレッドから呼ぶコードを書こうとしていた。

直し方は同じだった。「探す」と「RUNNING にして保存する」を1つの操作に
まとめて、1つのロックで囲む。20スレッドから同時に10個のタスクを奪い合わせる
テストを書いて、必ず10個ちょうど・重複なく確保されることを確認した。

## 今日の教訓

同じ形のバグに2回目に出会うと、直し方に迷わなくなる。1回目に気づいた
「採番と保存を分けない」という原則を、今度は自分で意識して探しにいった
おかげで、実際に手を動かす前に「ここも同じ形をしていないか」と疑うことが
できた。

もう1つの教訓は、**「仕様書に書いた」と「実装した」は別の状態**だという
当たり前のことだ。docstringには「Tier 1では」と正直に書いてあったのに、
自分では「そのうち直す」の「そのうち」を長い間読み飛ばしていた。設定ファイルの
コメントだけを見て「この数値を変えればスケールする」と信じ込んでいたら、
実機で並列化の効果がまったく出ないまま気づかずに運用していたと思う。
