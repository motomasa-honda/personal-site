## TL;DR

以前このブログで書いた、Mac miniとUbuntu機を直結してインターネットを共有する環境
(`192.0.2.1` / `192.0.2.2`)で、今度は逆方向——Ubuntu機からMac mini上で動くローカルLLM
サーバーへアクセスしようとして、TCP接続はできるのに応答が一切返ってこない現象にぶつかった。
2つの仮説を順番に試してどちらも外れた後、カーネルの接続状態を直接見ることで、ようやく
本当の原因にたどり着いた話を書く。

---

## 現象: 繋がるのに、何も返ってこない

Mac mini上でローカルLLMサーバー(Ollama)を動かしていて、これを直結しているUbuntu機からも
使いたかった。サーバー自体はすべてのネットワークインターフェースで待ち受ける設定に
なっていたので、単純に繋がるはずだと思っていた。だが実際にUbuntu機から叩くと、こうなる。

```
$ curl -m 5 http://192.0.2.1:11434/api/tags
(何も返ってこない)
* Empty reply from server
```

生のTCP接続だけを試すと、こちらは普通に成功する。

```
$ nc -zv 192.0.2.1 11434
Connection to 192.0.2.1 port 11434 [tcp/*] succeeded!
```

接続そのものはできる。だがHTTPリクエストを送ると、何も返ってこないまま切断される。

## 仮説1: バインド設定が間違っている → 外れ

真っ先に疑ったのは「サーバーがローカルホスト限定でしか待ち受けていないのでは」という
ことだった。だが設定を確認すると、既にすべてのインターフェースで待ち受ける設定に
なっていた。ここでいったん行き詰まった。

## 仮説2: NATが自分自身宛の通信を巻き込んでいる → これも外れ

このMac miniは、以前の記事で書いた通りインターネット共有の設定をしている
(Wi-Fi経由のインターネットを直結先のUbuntu機に流す)。この手のNAT設定は、共有先の
サブネットから来た通信を「インターネットへ出ていくもの」として一律で変換してしまい、
実は共有元(このMac mini自身)宛の通信まで巻き込んでしまうことがある——いわゆる
hairpin NATの問題だ。

この仮説はもっともらしく見えたので、共有先の自分自身のアドレス宛だけをNAT変換の対象から
除外する例外ルールを追加した。

```
no nat on en1 from 192.0.2.0/24 to 192.0.2.1
nat on en1 from 192.0.2.0/24 to any -> (en1)
```

だが適用して試しても、症状は一切変わらなかった。ここで2つ目の仮説も外れたことになる。

## カーネルの状態を直接見る

2つの仮説が外れた時点で、憶測でさらに設定を変えるのをやめて、実際に何が起きているのかを
カーネルのファイアウォール状態から直接確認することにした。

```
$ sudo pfctl -s state | grep 11434
ALL tcp 192.0.2.1:11434 <- 192.0.2.2:36376  TIME_WAIT:TIME_WAIT
```

`TIME_WAIT:TIME_WAIT`——これは接続が**ブロックされた**状態ではなく、**正常に確立されてから
きちんとクローズされた**ことを示す状態だ。もしファイアウォールやNATが接続を妨害していたなら、
こんな綺麗な終わり方はしない。

この一行で、疑う場所がネットワークの層からアプリケーションの層へ切り替わった。サーバー
自体は接続を正常に受け付けて、正常に閉じている。つまり——**サーバーは接続してきた相手を見て、
意図的に何も返さずに閉じている**、ということだ。

## 本当の原因: 接続元の許可リスト

ローカルLLMサーバーには、どこからの接続を信頼するかを制御する設定がある。見直すと、
このMac mini自身のループバックアドレスは登録されていたが、**直結しているUbuntu機側の
サブネットは最初から登録されていなかった**。バインド設定(どのインターフェースで待ち受けるか)
とは別に、接続元を検証する仕組みが別レイヤーとして存在していて、そこで弾かれていた。

```
OLLAMA_ORIGINS=http://192.0.2.1:*,http://192.0.2.2:*,<既存の localhost 等>
```

このサブネットの2台のアドレスを許可リストに追加してサーバーを再起動すると、Ubuntu機からの
リクエストは即座に本来のレスポンスを返すようになった。

## 今日の教訓

「繋がるのに応答がない」という症状は、ネットワーク層(ファイアウォール、NAT、ルーティング)
でも、アプリケーション層(接続は受け入れるが意図的に沈黙する)でも、同じように見える。
表面的に思いつく設定を順番に疑って直すやり方は、今回のように運が悪いと2連続で外れる。

`pfctl -s state`のような、カーネルが実際に記録している接続状態を早い段階で見ておけば、
「ブロックされているのか」「正常に終わった後に何も言ってこないだけなのか」を一発で
区別できた。これを先にやっていれば、1つ目の仮説を試す前に、そもそも疑う場所が
ネットワークではなくアプリケーション側だと分かっていたはずだ。次に同じ症状に出会ったら、
設定をいじる前にまずこれを見る。
