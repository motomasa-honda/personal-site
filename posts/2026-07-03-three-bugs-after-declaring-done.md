## TL;DR

段階を区切って進めてきた移行作業が全部終わり、テストも全部通った。ここで
「完了」と宣言する前に、もう一度全体を読み返してみた。3つ、実害のあるバグが
見つかった。どれもテストは通っていたのに見逃されていたもので、修正前のコードに
戻して「本当にテストが落ちるか」を確認してから直した。技術的に面白かったので、
3つとも書き残しておく。

---

## 1. 時系列に読み出す処理が、過去の記録を上書きし忘れる

ある処理の実行結果を、後から履歴として読み出せるように記録に残す仕組みを
作っていた。1回の実行につき複数の記録が残ることがある——例えば「失敗した」という
記録の後に、自動修正が効いて「今度は成功した」という記録が続く、というように。

読み出し側は、これらの記録を古い順に1つずつ処理して、最後に「結果はこうでした」を
組み立てる。ここでのミスは単純だった: 成功の記録を見た時に、それより前にあった
失敗の記録の情報(エラーメッセージ)を、消し忘れていた。

```python
for record in records:
    if record.is_success_marker():
        cv_score = record.value
        # ここで error_log をクリアし忘れていた
    elif record.is_failure_marker():
        error_log = record.error
```

結果、「最終的に成功した」という記録に、「実は途中で1回失敗していた」という
過去の痕跡がそのまま残ってしまう。これ単体では実害のない情報のゴミに見える。

問題は、この痕跡を別の仕組みが読んでいたことだった。「直近3回、同じエラーが
続いたら、これ以上続けても無駄なので止める」という安全装置が、この `error_log`
フィールドを見て判断していた。もし3回連続、同じ理由で1回失敗→自動修正→成功、
という同じパターンを繰り返すと——実際は3回とも成功しているのに、この安全装置は
「3回連続同じエラー」を検知して**ループを誤って止めてしまう**。

テストで再現させてみると、想定通りだった。修正前のコードに一時的に戻して、
「毎回同じ理由で1回こけて自動修正される」という状況を再現するテストを流したら、
本当に4回中3回目で誤って止まった。直したのは、成功を示す記録を見た瞬間に、
過去のエラー情報を明示的にクリアするという1行だった。

## 2. 採番と保存が別の操作だったせいで起きる競合

実験結果を記録する際、「この競技における次の試行番号は何番か」を調べてから、
その番号で保存する、という2段階の処理になっていた。

```python
next_no = store.next_trial_no(competition)  # ここでロックが外れる
trial = Trial(trial_no=next_no, ...)
store.save_trial(trial)                      # 別のロックで保存
```

それぞれの操作自体はスレッドセーフに書かれていたが、**2つの操作の間**は
ロックが外れている。今はまだ1つの処理しか同時に走らないので問題は起きないが、
将来、複数の処理を並行して走らせるようになった瞬間、こうなる:

処理Aが「次は5番」と調べる → 処理Bも(Aがまだ保存する前に)「次は5番」と
調べる → 両方とも5番として保存を試みる → 一意制約により、後から保存した方が
先に保存した方を**黙って上書きする**。1件分のデータが痕跡もなく消える。

直し方は、「番号を調べる」と「保存する」を1つの操作にまとめて、その1つの
操作全体を1つのロックで囲むこと。これで2つの処理が同時に来ても、片方が
終わるまでもう片方は順番待ちになり、番号が重複しない。8つのスレッドから
同時に保存を試みるテストを書いて、全部が別々の番号を得ることを確認した。

このバグは「今は起きていないが、将来の変更で確実に起きる」という種類のもので、
見つけるには「今の実装が前提としている並行性の制約」を疑って読む必要がある。
コードは正しく動いていたので、テストを流すだけでは絶対に見つからなかった。

## 3. プロセスをkillしようとしたら、もう死んでいた

外部プロセスを起動してタイムアウト管理をする処理で、制限時間を超えたら
プロセスグループごと強制終了する、という実装になっていた。

```python
except TimeoutExpired:
    os.killpg(os.getpgid(proc.pid), signal.SIGKILL)
```

ここには小さな競合状態がある。「制限時間を超えた」という通知が来てから、
実際に `killpg` を呼ぶまでの間に、もしプロセスが自然に終了していたら——
`os.getpgid` はもう存在しないプロセスグループを探そうとして例外を投げる。
この例外は捕まえていなかったので、そのままクラッシュしてしまう。

これは移植元のコードにも同じ形で存在していたバグで、気づかれずに残っていた
ということでもある。直したのは、その例外だけを明示的に捕まえて、
「もう死んでいるなら、それは成功(kill する必要がない)」として扱うこと。

```python
try:
    os.killpg(os.getpgid(proc.pid), signal.SIGKILL)
except ProcessLookupError:
    pass  # already gone — nothing to kill
```

3つの中でこれが一番地味だが、一番「発生確率は低いがゼロではない」タイプの
バグで、長時間運用して初めて踏む類のものだと思う。

## 今日の教訓

3つとも、個別に見れば小さい。でも共通しているのは、**「テストが通っている」は
「バグがない」の証明にはならない**ということだ。1つ目と2つ目は、そもそも
その状況を再現するテストを書いていなかったから見逃されていた。3つ目は、
起きる条件が稀すぎて偶然テストに引っかからなかっただけだ。

「実装が終わった」の後に、もう一度全体を読み返す時間を意図的に取ったから
見つかった。急いで次の作業に移っていたら、これらは本番相当の環境で初めて
表面化していたはずだ。特に1つ目は、安全装置そのものが誤動作するという
性質上、気づくのが遅れるほど「なぜか原因不明でループが止まる」という
デバッグしにくい症状になっていたと思う。
