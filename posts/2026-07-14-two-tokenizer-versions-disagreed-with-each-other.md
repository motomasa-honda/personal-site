## TL;DR

学習済みのLoRAアダプタを本番のベースモデルに統合し、GGUF形式に変換して
Ollamaへ取り込む——という一連のパイプラインを組んだ。統合(merge)は
成功したのに、次のGGUF変換の工程で、統合したばかりのモデルを読み込む
段階でいきなり失敗した。原因は、2つの工程が別々の環境で動いていて、
片方が保存したファイル形式を、もう片方が読めなかったことだった。

---

## 起きたこと

やろうとしていたのは3段階の作業だった。

1. LoRAアダプタ(ベースモデルの一部だけを追加学習したもの)を、
   ベースモデル本体に統合する
2. 統合済みのモデルを、Ollamaが読めるGGUFという形式に変換する
3. Ollamaに取り込んで、実際に動くモデルにする

1つ目の工程は、PEFTというライブラリ(LoRAを専門に扱うツール)が入った
環境で実行した。無事に成功し、統合済みのモデルのファイル一式(設定
ファイル、重みファイル、トークナイザ関連のファイル)がディスクに
書き出された。

続けて2つ目の工程。llama.cppという別のプロジェクトが配布している
変換スクリプトを、これも専用に用意した別の環境(1つ目とは意図的に
分けていた)で実行した。すると、こんなエラーで落ちた。

```
AttributeError: 'list' object has no attribute 'keys'
```

1つ目の工程が書き出した、まさにそのファイルを読み込もうとした瞬間の
エラーだった。統合自体はさっき成功したばかりで、ファイルは確かに
そこにある。

## なぜ2つの環境を分けていたか

これには理由があった。GGUF変換ツールが要求するライブラリ一式を、
LoRA統合に使っている環境にそのままインストールすると、GPU計算に
使っている別のライブラリのバージョンを巻き込んで壊してしまう
リスクがあった。以前に一度、無関係なはずのインストール作業で
GPU環境を壊しかけたことがあり、それ以来「性質の違う外部ツールは、
たとえ面倒でも別々の環境に隔離する」というのを徹底するようにして
いた。今回もその方針通り、統合用の環境と変換用の環境を、意図的に
別々に用意していた。

## 原因: 同じライブラリの、微妙に違うバージョン

エラーを検索してみると、ちょうど同じ現象を報告している記録が複数
見つかった。原因はこうだった。

モデルの設定ファイルの中には、そのモデル特有の特殊なトークン(文の
区切りなどを表す記号)を記録する項目がある。この項目の書式が、
使っているライブラリのバージョンによって違う。新しいバージョンは
リスト形式で書き、古いバージョンは辞書形式を期待して読む。

1つ目の工程(統合)の環境は新しいバージョンのライブラリが入っていて、
リスト形式でファイルを書き出していた。2つ目の工程(変換)の環境は、
変換ツール自体が指定していた、それより一段階古いバージョンが入って
いて、辞書形式だと決めてかかって読もうとしていた。同じ「トークナイザの
設定ファイル」という一枚の紙に、送り主と受け取り主で違う言語の
読み方を当てはめていた、という状態だった。

## 直し方: バージョンを揃える

直し方自体は単純だった。変換側の環境のライブラリを、統合側と
同じくらい新しいバージョンに上げた。

```bash
pip install -U 'transformers>=5.0'
```

これで変換が通り、実際に本物のGGUFファイルができ、Ollamaに取り込んで、
実際に応答が返ってくるところまで確認できた。

コード自体は何も直していない。直したのは環境の側だった。

## 今日の教訓

「同じ性質の作業をする2つの外部ツールを、意図的に隔離した環境に
分ける」というのは、正しい判断だったと今も思う。GPU環境を巻き込んで
壊すリスクを避けられたのは事実だ。

ただ、隔離するというのは「互いに知らないバージョンを平気で使い始める」
という新しいリスクを生む、ということもセットで覚えておくべきだった。
片方の環境で何のライブラリを何のバージョンで動かしているかは、
もう片方からは見えない。今回はたまたま「片方が書いたファイルを、
もう片方が読む」という受け渡しがあったから表面化したが、これは
「別々に隔離したはずの環境同士が、実は暗黙のうちにデータの受け渡し
契約を結んでいた」というのに気づいていなかった、ということでもある。

環境を分けるときは、分けた瞬間に「この2つの環境の間で、実際には
何が受け渡されるのか」を明示的に書き出しておく方がよかった。今回は
それをせず、エラーが起きてから慌てて調べる形になった。次に似た
構成(ある工程の出力を、別環境の別工程が読む)を作るときは、両方の
環境が同じ理解でその受け渡しを扱えるか、先に確認しておきたい。
