## TL;DR

- 前回「ensemble が CV で圧勝するのに LB で効かないのは探索の多様性不足だ」と結論づけた。今回その仮説を実機で検証したら、**仮説は外れていた**
- 検証の副産物で真犯人が見つかった。`oof_predictions/` (各 trial の out-of-fold 予測) が **run 跨ぎでクリアされず、数日前の trial が今回の ensemble に混入していた**
- 古い trial を積むほど、ensemble の貪欲最適化が **固定された OOF ラベルに overfit** する。CV だけ良くなって LB に転写されない = CV-LB ギャップそのものだった
- ジョブ開始時に OOF プールを消すだけで、**CV-LB ギャップが 0.0032 → 0.0009 に縮み、LB ベストが 0.12807 → 0.12594 に更新**された (House Prices)
- 「CV が悪化したのに LB は改善する」という overfit の教科書的シグネチャを実スコアで観測できた

## 仮説検証のはずが、別の犯人を踏んだ

前回の記事で、ensemble (hill climbing) が CV で勝つのに LB で動かない問題の真因を「ブレンドする中身が全部 GBDT で似すぎているから」と推定し、探索方向に異系統モデルを足した。今回はその end-to-end 検証 — 異系統 trial が実際にブレンドされて LB を更新するか — をやるつもりだった。

ところが検証ログを精読すると、おかしな点に気づいた。今回の ensemble が混ぜている trial の中に、**今日生成していないもの**が混じっていた。

```
[hill_climb] candidates: [3, 112, 1, 2, 132, 121, 110, 122, 131, 111]
[hill_climb] step 0: +trial_3   score=0.12799
[hill_climb] step 1: +trial_121 score=0.12565   # ← trial_121 は数日前の run のもの
[hill_climb] step 2: +trial_1   score=0.12515
[hill_climb] step 3: +trial_132 score=0.12476   # ← trial_132 も別 run のもの
```

trial 番号と保存ファイルのタイムスタンプを突き合わせると、`trial_121` も `trial_132` も**数日前の別 run で作られた OOF ファイル**だった。ブレンドの中身は今日の探索結果ではなく、過去の遺産の寄せ集めだったのだ。

## なぜ古い OOF が混ざると CV-LB ギャップになるのか

ワークスペースの `oof_predictions/` には、trial ごとに「学習データに対する out-of-fold 予測」が `.npy` で溜まっていく。hill climbing はこの OOF を読み込み、**OOF 上の CV スコアを最小化するように**重みを貪欲に決める。

問題は、このディレクトリが **run のたびにクリアされていなかった**こと。同じコンペを何度も回すと、過去 run の trial が消えずに残り、新しい run の hill climbing がそれも候補に入れてしまう。

ここで効いてくるのが overfit の構造だ。

- OOF のラベル (正解) は固定。学習データの真値だから run をまたいでも変わらない
- hill climbing は「その固定ラベルに対する CV」を貪欲に下げる
- 候補 trial が増えれば増えるほど、**たまたまその固定ラベルにフィットする組み合わせ**が見つかりやすくなる
- それは OOF (= CV) には効くが、未知の test (= LB) には効かない

つまり古い trial の積み増しは、**CV を実際の汎化性能から乖離させる装置**として働いていた。さらに run ごとに特徴量パイプラインが変われば、OOF が指す前提条件も揃わなくなる。私たちがずっと追っていた「CV だけ良くて LB が動かない」現象の、少なくとも一因はこれだった。

## 修正は一行 — ジョブ開始時にプールを消す

ジョブ投入時のクリーンアップに、OOF ディレクトリの削除を足すだけ。

```python
# 古いファイルをクリーンアップ
for f in ["main.py", "submission.csv", "metrics.json"]:
    p = workspace / f
    if p.exists():
        p.unlink()

# OOF プールは run 跨ぎでクリアする。残すと前回 run の trial が
# hill_climb / stacking に混入し、固定 OOF ラベルへの overfit
# (CV だけ良く LB に転写されない CV-LB ギャップ) を生む。
import shutil
oof_dir = workspace / "oof_predictions"
if oof_dir.exists():
    shutil.rmtree(oof_dir, ignore_errors=True)
```

run 内のイテレーション間では OOF が溜まってよい (同じ run の trial を混ぜるのが ensemble の狙いだから)。クリアするのは **ジョブの開始時だけ**。これで「1 run = 1 つの綺麗な評価プール」になる。

## 結果: CV は悪化、LB は改善

同じ設定で、プールが汚れた状態と綺麗な状態を比べた。

| プール | ブレンドの CV | public LB | CV-LB ギャップ |
| --- | --- | --- | --- |
| 古い trial 混入 | 0.12476 | 0.12797 | 0.0032 |
| クリーン | 0.12505 | **0.12594** | **0.0009** |

注目すべきは、**クリーンな方が CV は悪い (0.12505 > 0.12476) のに LB は良い (0.12594 < 0.12797)** という逆転だ。これは overfit の教科書的なシグネチャそのもの — 汚れたプールは見かけの CV を盛っていただけで、その上げ幅は test に 1 ミリも転写されていなかった。綺麗なプールの「正直な CV」の方が、ちゃんと LB に乗る。

CV-LB ギャップは 0.0032 → 0.0009 に縮み、House Prices の LB ベストは更新された (このあと別の要因も重なって最終的に 0.12255 まで伸びた)。

## 学び

- **「CV が上がった」を疑え**。ある修正で LB も更新したが、中身を見たら過去の trial の寄せ集めで、仮説は何も検証できていなかった。スコアだけ見ると間違った結論に着地する
- **評価系の汚染は最強のバグ**。モデルでもハイパラでもなく、「評価に使うデータがいつのまにか汚れている」のが CV-LB ギャップの正体だった。固定ラベルに対して候補を積み増す最適化は、必ずそのラベルに overfit する
- **CV が悪化して LB が改善したら勝ち**。直感に反するが、汎化を測れている証拠。逆 (CV だけ改善) は警報
- 当初の仮説 (多様性不足) を検証しに行って別の真因を踏んだ。**狙った魚は釣れなくても、網に別の大物がかかることがある**。検証は仮説の真偽だけでなく、想定外を観測する場でもある
