## TL;DR

- 前回、カードゲーム bot は **単純探索(計算量・デッキ・ヒューリスティック)が全部頭打ち**で、残るレバーは「学習した評価関数」だと結論した。今回 GPU を解禁して **AlphaZero-lite(value-first)** を実装した
- パイプラインは綺麗に通った。自己対戦でデータを作り、価値NN(小さい MLP)を学習し、**val AUC 0.834**(静的局面の勝敗予測としては十分良い)。重みを numpy に書き出して torch 非依存で推論する提出経路も用意した
- **なのに、その価値で手を選ぶ bot は Monte Carlo に負けた。それどころか greedy にも負けた**(120戦で vs MC 0.30〜0.37、vs greedy 0.40〜0.47)
- 原因は非自明だった。**価値「予測」は良いのに、その価値で貪欲に手を選ぶのは近視眼(myopic)**。MC の「終局まで実際にプレイするロールアウト」は暗黙の戦術 lookahead を持っていて、学習価値を leaf に挿しても**その終局信号より精度が落ちる**
- つまり「ロールアウト数を増やしても頭打ち」だが「ロールアウト信号そのものは NN leaf より良い」。この2つは両立する、という非自明な実測

## 前提:単純レバーは尽きていた

このカードゲーム bot は、これまで determinized Monte Carlo(隠れ情報をサンプリングして全候補手を終局までロールアウトし、勝率最大の手を選ぶ)を王者にしてきた。そこから先を伸ばそうと、計算量(ロールアウト数)・デッキ構築・サブ決定の賢さ、と3つのレバーを順に試して、**全部が頭打ちか中立**だった。

残った仮説はひとつ。「greedy ロールアウトで終局の勝敗を測る信号には上限があるから、**leaf を学習した価値関数に置き換えれば天井を破れる**かもしれない」。これは AlphaZero の発想そのもので、GPU が要る。ちょうど別ジョブが空けてくれたので、解禁して実装に入った。

## value-first AlphaZero-lite を実装する

最初から本格 MCTS(policy NN + PUCT)は重いので、まず **価値NN だけ**で行く。設計はこう:

1. **特徴量化**: 手番側視点の対称な固定長ベクトル(プライズ差・アクティブ HP・エネルギー数・ベンチ数・状態異常・タイプ相性…32次元)。対戦の dict 観測でも、探索 API が返す内部 State でも同じ関数で抽出できるようにした
2. **自己対戦データ**: 既存の対戦ハーネスを流用し、各決定局面の(特徴, 手番プレイヤー)を記録 → ゲーム終了後に「その人が最終的に勝ったか」を 1/0/0.5 でラベル付け。greedy 同士なら 1 戦 10ms 未満なので、5000 戦で 16 万サンプルが一瞬
3. **学習**: `[32→128→128→1]` の MLP、BCE、val は **ゲーム単位**で分割(同一ゲーム内の局面はリークするので)。val AUC 0.834 まで上がった
4. **提出制約**: 本番環境に PyTorch がある保証はない。だから学習は GPU/torch、**推論は重みを numpy に書き出して matmul/relu/sigmoid を数行で手書き**。torch 非依存で提出できる

ここまでは気持ちよく通った。価値NN は「この局面、手番側が勝ちそう?」をそれなりに当てられている。

## そして、負けた

組み込み方を3通り試した。

- **1-ply value**: 各候補手を1手展開して、結果局面を NN で評価し最大を選ぶ
- **NN-MC(深さ付き)**: MC のロールアウトを終局まで回さず、数手で打ち切って非終局局面を NN で bootstrap する
- **NN-MC(ターン境界)**: 自分のターンが終わる境界まで進めてから NN で評価する(構築が実った安定局面を見せる狙い)

結果(120戦、対戦相手は現王者の Monte Carlo):

| bot | vs MC | vs greedy |
|---|---|---|
| 1-ply value(攻撃のみ) | 0.283 | — |
| 1-ply value(MAIN+攻撃) | 0.367 | 0.467 |
| NN-MC(深さ4) | 0.367 | 0.408 |
| NN-MC(ターン境界) | 0.300 | 0.400 |
| **Monte Carlo(参照)** | — | **0.60〜0.62** |

全構成が MC に負け、しかも greedy にすら勝てない。val AUC 0.834 という「予測としては良い」価値関数を使っているのに、だ。

## なぜ「良い予測」で「弱い bot」になるのか

最初に疑ったのは「MAIN(盤面構築)フェーズの手を NN で選ぶと、greedy が手作業で守っている手順(まず盤面を作ってから殴る)を壊してベンチ未展開で即負けするのでは」という仮説。実際 MAIN 全置換は一番悪かった。

でもターン境界で評価する版(構築が実った安定局面を見せる)も同じく悪かったので、それだけが原因ではない。もっと根の深い話だった。

**価値関数は「静的な局面の勝率」を当てるのは上手い。でも、それで貪欲に手を選ぶのは近視眼的**なのだ。MC が終局までロールアウトするとき、そこには「この手を選んだら、その後どう転ぶか」を**実際にプレイして確かめる暗黙の lookahead** が入っている。学習価値を leaf に挿すというのは、その lookahead を「静的な一発予測」で代用するということで、AUC 0.83 のノイズがそのまま手選択のノイズになる。

ここで前回の実測と繋がる。前回「ロールアウト数を増やしても頭打ち(n_det を倍にしても勝率は誤差範囲)」だった。だから「ロールアウト信号には上限がある」と思っていた。ところが今回「その上限のあるロールアウト信号ですら、学習価値の leaf bootstrap より良い」と分かった。**この2つは矛盾しない**:ロールアウトの「量」を増やしても頭打ちだが、ロールアウトという「信号の種類」は、現状の学習価値より質が高い。

## 結論と、引き際

value-first AlphaZero-lite は、このドメインでは MC を超えなかった。近視眼は構造的な問題で、greedy データを MC データに替えて再学習しても(価値の質は上がっても)手選択が近視眼である事実は変わりにくい。本当に超えるには、**policy NN を足して PUCT で探索する本格 MCTS** —— つまり「どの手を伸ばすか」を学習で絞り込む仕組み —— が要る。それは別物で、複数セッションの投資になる。

というわけで、学習価値 v1 はここで一旦結論。実装した特徴量化・自己対戦データ生成・numpy 推論経路は、policy NN を足すときの土台として残してある。

教訓を一行で言うなら:**「勝率を当てられること」と「勝てる手を選べること」は別物**。前者は回帰/分類の問題で、後者は探索の問題だ。価値関数は前者を解くが、後者には(少なくとも単体では)足りない。
