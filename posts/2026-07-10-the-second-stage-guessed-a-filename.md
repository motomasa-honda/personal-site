## TL;DR

分析タスクを複数の工程(データ読み込み→特徴量作成→学習、のような)に
分割して、それぞれ別々にコードを書かせて実行する仕組みを作った。設計は
紙の上ではきれいに閉じていたが、実際に本物のモデルへ2工程分のコードを
書かせてみたところ、2つ目の工程が「1つ目の工程が保存したはずのファイル」を
毎回間違った名前で探しに行き、失敗し続けた。原因と直し方、そして
「設計レビューでは絶対に見つからなかった」という話を書く。

---

## 起きたこと

やろうとしていたのは単純なことだった。「データを読み込んで前処理する」
「前処理済みのデータを読み込んでモデルを学習する」という2工程のタスクを、
それぞれ別のプロセスとして順番に実行する。1工程目の出力ディレクトリを
2工程目に「ここに前の工程の結果があります」と伝えれば、後は生成AIが
よしなに読み書きしてくれるはずだった。

1工程目は問題なく動いた。CSVを読み込み、特徴量を1つ足し、次のように
自分で決めたファイル名で保存した。

```python
train_df.to_csv('train_engineered.csv', index=False)
```

2工程目に「前工程の出力は `../load/` 以下にあります」とだけ伝えて
コードを書かせたところ、返ってきたコードはこうだった。

```python
df = pd.read_csv('../load/processed_data.csv')
```

`processed_data.csv` などというファイルは存在しない。1工程目が実際に
書いたのは `train_engineered.csv` だ。実行結果は当然
`FileNotFoundError`。何度試しても同じで、生成されるファイル名の予測は
毎回それらしいが、毎回外れた。

## なぜ外れたか

原因はシンプルだった。2工程目にコードを書かせる時点で、1工程目の
コードは**まだ実行されていなかった**。

もともとの実装は、2つの工程のコードを両方とも実行前に一括で生成していた。
そうすれば2つのタスクを並行して投げられて速いだろう、という発想だった。
だが2工程目に渡していた情報は「前工程はこういう処理をします」という
自然文の説明だけで、前工程が実際にどんなファイル名で何を保存するかは、
1工程目のコード生成担当が実行時にその場で決めることであって、事前には
誰にも分からない。2工程目の担当は、もっともらしい名前を**当てずっぽうで**
埋めるしかなかった。

これは既視感のある失敗だった。以前、別の場面で「渡したはずのデータを
無視して、AIが勝手にもっともらしいデータを補ってしまう」という問題に
遭遇したことがある。今回起きたのは、その一段先の話だった。データその
ものではなく、**前の工程の出力という「事実」**を、後工程のAIに渡す
手段がなかった。手段がなければ、AIは自然にそこを推測で埋める。

## 直し方: 実行してから、次を書かせる

修正は「並行して速くする」という最初の発想を一部あきらめることだった。
工程を依存関係の層に分け、ある層を実際に実行してから、次の層のコードを
書かせるように順序を変えた。

```python
real_files = list_node_output_files(workspace_root, "load")
# -> ["train.csv", "train_engineered.csv", "metrics.json", ...]

description += (
    f"'../load/' contains exactly these files "
    f"(use these exact names, do not guess a different filename): "
    f"{', '.join(real_files)}"
)
```

前の工程が実際にディレクトリへ残したファイル名を機械的に列挙し、次の
工程のプロンプトにそのまま埋め込む。推測させず、事実をそのまま渡す。

修正後、同じ2工程を実機の本番モデルへもう一度投げた。今度は
`train_engineered.csv` という正しい名前で読み込み、学習も最後まで
完走し、実際の精度スコアも出力された。同じ依存関係にある工程同士は
これまで通り並行実行できるので、速さを完全に失ったわけではない
——「前工程の結果を必要とする工程」だけが、律儀にその工程の完了を
待つようになっただけだ。

## 今日の教訓

設計を紙の上でどれだけレビューしても、この種のバグは見つからなかった。
「前工程の出力ディレクトリを教える」というアイデア自体は正しく、
テストコードを書いて動作確認もしていた。だが自動テストで使っていた
モデルはあらかじめ用意した固定の返答を返すだけの偽物で、「本物のAIが
実際に何を書くか」という一番肝心な部分は、最後まで検証していなかった。

本物のモデルに実際にコードを書かせて、実際に実行してみて、初めて
「ファイル名は事実であって、説明文からの類推対象ではない」という
当たり前のことに気づいた。パイプラインの前後の工程をまたいで何かを
伝えるときは、それが解釈や指示なら言葉で渡してよいが、実行環境の
具体的な状態(ファイル名、パス、実際に存在するもの)は、必ず後工程が
実行される直前に、その場で機械的に確認してから渡す。前回の教訓と
同じ形をしていたが、適用範囲が一段深いところにまで及んでいるとは
思っていなかった。
