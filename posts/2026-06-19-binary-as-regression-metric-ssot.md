## TL;DR

- 自律 Kaggle システムを、初めて **本番のメダル系コンペに近い実データ**(終了済 Playground Series, F1 の「次ラップでピットするか」を当てる二値分類・AUC)で end-to-end 回した
- スコアは出た。公開 LB **0.94145** / private **0.94268**、3023 チーム中 ~2300 位。中央値(0.948)を僅かに下回る位置。**初めて train/test を未知の実データで完走した**こと自体が収穫
- ところがログを解剖したら、**二値分類タスクが丸ごと『回帰』として解かれていた**。原因は拍子抜けするほど単純で、ターゲット列 `PitNextLap` が `0`/`1` ではなく **`0.0`/`1.0`(float)** で入っていたこと
- さらに非自明な落とし穴があった。**この回帰誤判定を素直に直すと、スコアはむしろ悪化する**。分類テンプレが AUC コンペでも**ハードラベル(0/1)を提出**するバグを抱えていて、たまたま「回帰で連続値が出た」のと相殺して 0.94 になっていた
- 結論:バラバラに task を推測していた各部品を、**『コンペ指標を単一の真実(SSOT)にする』**よう一本化した。これは全表データコンペに効く恒久修正

## 実データで回して初めて見えるもの

このシステムは、これまで Getting Started(Titanic / House Prices)でしか通しを回していなかった。「作る」フェーズはほぼ終わっていて、足りないのは「**未知の実データで本当に動くか**」の検証だった。

そこで終了済みの Playground Series(後から提出してもスコアは出る)を選び、URL を投げるだけの自動実行に乗せた。指標の自動検出 → データ DL → 反復学習 → アンサンブル → 提出まで、6 イテレーションが ~30 分で完走。公開 LB 0.94145。**初めて、自分が一度も見ていない実データで end-to-end が通った**。

ただ、順位は中央値の少し下。そして LB が異常に密集していた:中央 0.948、トップ 0.955、25 パーセンタイル 0.942。**AUC を 0.007 動かすだけで順位が中央を大きく超える**。つまり「小さなスコア差」が「大きな順位差」になる。伸びしろがどこにあるか、ログを解剖した。

## 二値分類が、回帰として解かれていた

提出ファイルの予測値を見て違和感があった。AUC 用なのに、値が **負(-0.35)から 1.27** まで散らばっている。確率(0〜1)ではない。これは回帰モデルが 0/1 を連続値として予測した出力だ。内部記録を見ると、タスク種別が `tabular_regression`、確定版モデルの「CV」が 0.273(これは AUC ではなく RMSE 系の値)。

おかしい。指標は明確に AUC で、コンペ URL のパーサは正しく「AUC → 分類」と判定していた。なのに実際の学習は回帰で走っている。

原因はタスク種別を推定するヒューリスティックにあった。ざっくりこういう判定をしていた:

```python
if "float" in target_dtype or target_nunique > 50:
    return "tabular_regression"   # 連続ターゲット
if target_nunique >= 2:
    return "tabular_classification"
```

ターゲット列 `PitNextLap` は中身こそ 0 と 1 の二択だが、**dtype が float(`1.0` / `0.0`)**だった。`"float" in target_dtype` が**ユニーク数(=2)の判定より先に**当たって、二値分類が「連続ターゲット=回帰」に化けていた。0/1 を float で配るデータセットは珍しくないのに、dtype を先に見たせいだ。

しかも上流では、URL パーサが指標(AUC)から「これは分類」という**正解を既に持っていた**のに、解析ノードがそれを使わず、このデータ・ヒューリスティックの結果で上書きしていた。**一番確かな信号(指標)を、一番あやふやな信号(dtype)で潰していた**わけだ。

## 素直に直すと、悪化する

「じゃあ float 二値を分類に倒せばいい」—— そう思って分類経路を覗いたら、もっと厄介なバグが寝ていた。二値分類テンプレの提出部分がこうなっていた:

```python
pred_label = (pred > 0.5).astype(int)   # 確率を 0/1 のハードラベルに
sub = pd.DataFrame({id_col: ids, target: pred_label})
```

**AUC コンペなのに、ハードラベル(0/1)を提出している。** AUC はランク(予測の大小関係)で測る指標なので、0/1 に丸めるとランク情報が全部消えて、スコアは劇的に落ちる。OOF も CV も確率で計算しているのに、最終提出だけラベル化していた —— **CV と提出物のスコアが構造的にズレる**サイレントなバグだ。

ここで全体像が見えた。今回 0.94 が出ていたのは、**2つのバグが偶然キャンセルしていた**からだ:

- 回帰として解いた → 連続値が出た(AUC のランクには都合が良い)
- もし「正しく」分類経路に載せていたら → ハードラベル提出で AUC が崩れていた

つまり、片方だけ直すと相殺が崩れて悪化する。**偶然の上に乗ったスコア**だった。

## 指標を単一の真実(SSOT)にする

個別の if 文をパッチして回るのは筋が悪い。根本は「task 種別・提出形式・CV 採点・指標方向・確定版の指標・アンサンブルの指標・知識記録、が**各自バラバラに指標を推測していた**」ことだ。だから一箇所に集約した:

```python
def metric_policy(metric: str) -> dict:
    # 指標(Kaggle API 由来・最も確か)を唯一の真実にする
    # auc      -> {分類, 大きいほど良い, 確率提出}
    # logloss  -> {分類, 小さいほど良い, 確率提出}
    # accuracy -> {分類, 大きいほど良い, ラベル提出}
    # rmse     -> {回帰, 小さいほど良い, 値提出}
    ...
```

これを task / 方向 / 提出形式の起点にして、

1. **task は指標から確定**(指標で決まらない時だけデータ・ヒューリスティックに落とす)
2. **提出形式も指標から確定**(AUC・logloss → 確率、accuracy・F1 → ラベル、回帰 → 値)。テンプレの二値提出を `(pred>0.5)` 固定から指標準拠に
3. データ・ヒューリスティック側も保険として直す(**ユニーク数を dtype より優先**。float でも少数クラスは分類)

ライブのログで、再実行の解析がこう吐くようになった:

```
metric_policy: task=tabular_classification higher=True submission=proba
```

回帰誤判定は消え、確率提出に揃った。これは s6e5 固有の話ではなく、**0/1 を float で配る二値分類すべて**(Playground 系に多い)に効く恒久修正だ。

## おまけ:強い戦略が、些細なクラッシュで死んでいた

ついでに見つけた別の傷。最終イテレーションで生成された戦略が `lightgbm + target_encoding`(この手のコンペで定石の強い手)だったのに、`X.iloc[tr]` を NumPy 配列に対して呼んで `AttributeError` で即死していた。生成コードが途中で `X` を配列化していたのに、CV 分割で DataFrame 前提の `.iloc` を使ったせいだ。

LLM 生成コードの非決定的なミスなので、コード整形パスに**決定的な矯正**を足した:`var.iloc[idx]` の単純な添字だけを「DataFrame でも配列でも動くアクセサ」に置換する(スライスや属性チェーンには触れない)。これで強い戦略が些細な型不一致で落ちなくなる。

## 教訓

実データで一度通すだけで、Getting Started では絶対に出てこないバグが 3 つ落ちてきた。`0` と `0.0` の違い。AUC にハードラベルを出す盲点。生成コードの型不一致。どれも**未知の実データに当てて初めて顔を出す**やつだ。

一行でまとめるなら:**「正解(指標)を一番確かな信号として、末端まで一貫して流す」**。task も提出形式も CV も、各部品に推測させず、指標という単一の真実から導く。AutoML を名乗るなら、ここを曖昧にしてはいけなかった。
