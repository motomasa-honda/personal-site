## TL;DR

Docker で初めて実際にコンテナを起動して、Approve ボタンを本当にクリックしてみた。
そしてその過程で、これまで別々に育ててきた2つのプロジェクト——AI 組織をシミュレートする
「KRS-Nexus」と、Kaggle 攻略のために積み上げてきた知識基盤——の関係を、一日のうちに
2回作り直すことになった。

1. Docker のライブ検証で実バグを1つ発見・修正(依存宣言漏れ)
2. 「空の画面が表示できた」で満足しかけたのを指摘されて、実データで Approve ボタンを
   本当に押すところまでやり直した
3. アーキテクチャ設計を2回作り直し、最終的に「KRS-OS は KRS-Nexus という土台の上で
   動く独立したレイヤー」という3層構造に落ち着いた
4. その最初の1段(知識コネクタ)を実装・実データで検証・コミットまで完了

---

## 1. 「動いた」と「検証した」は違う

Docker Desktop を初めて入れて `docker compose up --build` を叩いた。ビルドは通ったが、
コンテナは起動直後にクラッシュした。

```
Error response from daemon: failed to create task for container: ...
exec: "uvicorn": executable file not found in $PATH
```

原因はシンプルで、`pyproject.toml` の依存リストに `uvicorn` そのものが一度も
書かれていなかった。`Dockerfile` の `CMD ["uvicorn", "main:app", ...]` は、
書かれた瞬間から一度も実際に実行されたことがなかったということだ。テストは全部通っていたし、
コードレビューも何周かしていたのに、「実際にコンテナを起動する」という一番単純な検証を
省略していたせいで、こんな基本的な抜けが何週間も見逃されていた。

依存を追加してテストを流し直し、コンテナは正常に起動した。API を叩いても 200 が返る。
ブラウザで開いても画面は出る。ここで「Phase A(実地検証)完了」と報告しかけた。

画面には「No in-flight Tasks.」とだけ表示されていた。

これは指摘されて初めて気づいたのだが、**空の画面が正しく表示されることは、何の検証にも
なっていない**。承認待ちのデータが1件もないのだから、承認ボタンが本当に機能するかどうかは
まだ何もテストしていない。「動いているように見える」と「動作を検証した」の間には、
思っていたより大きな溝があった。

やり直した。API 経由でテストデータを作る手段がそもそも用意されていなかった(読み取りと
承認決定しかできない設計になっている)ので、コンテナの中に直接入って、ストア層を叩いて
承認待ちの候補を1件差し込んだ。それが API 経由で見えることを確認し、実際にブラウザで
Approval Queue を開いて、**Approve ボタンを本当にクリックした**。ステータスが変わって
リストから消えたのを見て、ようやく「ブラウザからの操作が、UI・API・内部ロジック・永続化まで
一気通貫で動く」ことを確認できた。

## 2. 一枚の設計図が引き起こした2回の作り直し

数ヶ月前に自分で描いた設計図(認知ルーター:重い判断は Claude、繰り返しの多い判断は
ローカルLLM に振り分ける、というアーキテクチャ)を見返していて、これが今作っている
「AI組織シミュレーションシステム」とは実は別物だと気づいた。同じプロジェクト名で
呼んでいたが、中身は違う設計思想だった。

最初はこう考えた。「認知ルーターの設計は、AI組織の中で働く1人のエージェントの
内部実装として隠してしまえばいい」。組織の骨格はそのまま、社員1人の頭の使い方として
組み込む案だ。これなら両方とも作り直さずに済む。

しかしこれを説明したところ、「参照する形にしたくない」という指摘を受けた。片方が
もう片方を内部に隠して使う関係は、結局のところ「呼び出す側」と「呼び出される側」の
非対称な依存でしかなく、「1つのシステム」にはならない。

いくつかの案を検討した末——知識基盤側を本体にする案、完全に切り離す案、AI組織側を
土台にする案——最終的に落ち着いたのはこの形だった。

```
土台 (実行基盤: イベント駆動カーネル・タスクキュー・LLMルーター・組織管理・ツール層)
  └─ 認知アーキテクチャ (Heavy/Workhorse 振り分け・知識棚・シャドー評価)
       └─ 積み上げてきた実行ロジックが、将来的にここへ移行する
```

ポイントは、認知アーキテクチャを「土台の中に隠れた1機能」ではなく、**土台の上で動く
独立した名前を持つ製品**として扱うことだ。土台は実行基盤に徹し、その上に「どう考えるか」
という層を1枚載せる。これなら両者は対等ではないが、少なくとも「隠して呼び出す」関係では
なくなる。

大きかったのは、これまで別に運用してきた実行ロジック(コンペを選び、コードを書き、
提出するまでの一連の判断ループ)を、最終的にはこの新しい土台の上に移行させる、と決めた
ことだ。これは当初考えていたより踏み込んだ決断で、移行が終わればもとのリポジトリは
要らなくなる。ただし——移行には数週間規模の作業が要る。だから**移行が実戦で証明されるまで、
今動いている方は止めない**というルールも同時に決めた。積み上げてきた実績を、
まだ検証されていない新しい基盤のために止める理由はない。

## 3. 最初の1段 — 知識をどう引き渡すか

4段階の移行計画のうち、外部の認証情報が要らず今日中に着手できるのが最初の1段
(知識の接続)だった。数百件ある技術ノート(YAML frontmatter 付きの Markdown、
それぞれ「カテゴリ」「使用モデル」「対象コンペ」「品質スコア」を持つ)を、
土台側の知識グラフに繋ぎ込む。

土台側には既に「任意のディレクトリの `.md`/`.txt` を再帰的に読む」汎用コネクタが
あった。これをそのまま向ければ動く。だが実際に既存の抽出ロジックを読むと、
Markdown の見出し階層だけを見る設計になっていた(決定論性を保つための意図的な制約)。
frontmatter は完全に無視される。せっかく数百件分の構造化データがあるのに、
「概要」「適用条件」みたいな見出しの topic ノードしか出てこない。

frontmatter を活かすための拡張を考えたとき、一番避けたかったのはグラフのスキーマに
新しいフィールドを増やすことだった。ノード・エッジという既存の関係モデルは十分に
表現力があるはずで、フィールドを増やすと将来の互換性コストになる。試しに全部
既存スキーマの範囲でマッピングしてみた:

- 品質スコアは、ノードの「確信度」フィールドにそのまま乗る(意味的にも一致している)
- カテゴリ・使用モデル・対象コンペは、それぞれ専用の概念ノードを作り、
  「〜に分類される」「〜を使う」「〜に適用される」という種別付きのエッジで繋ぐ

これでスキーマ変更ゼロのまま、frontmatter の情報を失わずに取り込めた。

実際の Vault(1030件のアクティブなノート)に対して走らせてみると、3925ノード・
2895エッジが生成され、カテゴリは12種類、対象コンペは21種類、使用モデルは125種類に
正しく集約された。同じカテゴリを参照する複数のノートが、ちゃんと同じ1つの概念ノードに
収束している。

```python
distinct categories: 12 ['category:agent', 'category:cv', 'category:data_augmentation', ...]
distinct competitions: 21
distinct models: 125
```

## 4. 今日の教訓

- **「画面が出た」は検証にならない。** 空の状態で満足しかけた自分に、後から
  ヒヤッとした。実データを1件でも通してみないと、何も確かめたことにならない。
- **同じ名前で呼んでいた2つのものが、実は別の設計思想だった、という事故は起こる。**
  気づいたら、どちらかに寄せて済ませず、関係性そのものを言語化し直す価値がある。
- **スキーマを増やす前に、既存の関係モデルで表現できないか一度疑う。** 今回は
  ノード・エッジの組み合わせだけで、新しいフィールドを1つも増やさずに済んだ。
- **大きな移行を決めても、今動いているものを人質にしない。** 検証が終わるまでは
  並走させる、というルールを先に決めておくと、意思決定の重さに引っ張られずに済む。

一日のうちに設計方針を2回作り直すのは効率が悪いように見えるかもしれないが、
「参照関係にしたくない」という一言のおかげで、より良い形にたどり着けた。
次はこの土台の上に、認知アーキテクチャそのものを実装する番になる。
