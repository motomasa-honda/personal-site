## TL;DR

playground-series-s6e6、締切直前まで Public LB 0.97265 / rank 14 (Gold 境界線) にいた。
締切後 Private LB が確定して見たら **0.97002 / rank 405 (2309 チーム中)**。Bronze (top 10%
≈ 231 位) にも届かなかった。

| | Public | Private | 差 |
|---|---|---|---|
| 攻め (公開資産 + row-flip 系) | 0.97265 | **0.97002** | -0.00263 |
| 保険 (自前 CV で検証した genuine ensemble) | 0.97012 | 0.96952 | -0.00060 |

スコアの下落幅自体は事前に見積もっていた通りだった。想定外だったのは **rank の落差**。
順位が数十落ちる程度を覚悟していたが、実際は 14 位から 405 位まで丸ごと圏外に落ちた。

## 何をしていたか

このコンペは Public LB が test data の約 20% だけで計算され、最終順位は残り 80% の
Private LB で決まる仕様だった。序盤からこの構造には気づいていて、「公開資産をベースに
した row-level の flip 手法 (historical Public LB score を教師信号にして、個別行のラベル
を反転させる手法) は public-label probing になりやすい」という community の議論も読んでいた。

なので Final の 2 提出は意図的に分けた:

- **攻め**: 公開されている高スコア submission (ある作者が公開していた 0.97258 / 0.97259
  の完成品) を土台に、複数の公開 score-named CSV を教師信号にした ridge flip で数行だけ
  上乗せしたもの
- **保険**: 15 モデルの OOF を balanced_accuracy で重み最適化した、ridge flip を一切
  使わない自前の ensemble (nested cross-validation で +0.0026 の genuine improvement を
  確認済み)

「攻めが仮に public-probing で崩れても、保険が CV 通りに動いて Silver は確保できる」
という設計のつもりだった。

## 実際に起きたこと

保険は狙い通りだった。Public 0.97012 → Private 0.96952、差は -0.0006 でノイズの範囲。
**CV で検証した改善は、ちゃんと Private でも再現した**。ここは自信を持てる。

問題は攻めの方だった。Public → Private の**スコア差自体は -0.0026 で、これも事前予測の
範囲内**。でも rank の落差が全く違った。

土台にした公開 dataset (ある作者の完成品) 自体が、既に Public LB のフィードバックで
作り込まれたものだった。さらに「score-named CSV を教師信号にした row-flip」という手法
自体が、このコンペのコミュニティで相当流行していた。community の投稿を見返すと、複数の
高評価 public kernel が同じアプローチを採っていた。

つまり Public 上位 14〜100 位あたりは、**同じ 20% のスライスに対して、同じロジックで
似たような overfitting をしていた集団**だったんだと思う。Private (残り 80%) では、その
集団がまとめて崩れた。個々人が数十位ずつ落ちるという生易しい話ではなく、集団ごと
シャッフルされて全然違う位置に散らばった。

## 数字で見る誤算

コミュニティの投稿にこういう分析があった:

> public top-10 の観測 spread がノイズフロアより低い → Public top-10 のうち Private
> top-10 に残るのは平均 1 人、30% の確率で誰も残らない

これは読んでいたし、「シビアだな」とは思っていた。でも自分の頭の中では「1 人しか
残らない」を「1 位以外はだいたい 2〜3 位落ちる」くらいの温度感で解釈していた。実際は
「1 位だけ残って、他は文字通り桁が変わる位置まで飛ぶ」という規模だった。この解像度の
違いが今回の見積もりミスだったと思う。

## 教訓

**公開されている高スコアの完成品を土台にする時点で、既にリスクを継承している。**
出どころのわからない高スコア CSV は、それ単体が Public LB フィードバックの産物である
可能性を疑うべきだった。

**流行している手法は、個性ではなく集団リスクになる。** row-flip 系のような手法が
コミュニティで広く使われている場合、差別化どころか、同じ overfitting を集団で共有する
ことになる。shake-up が来た時、その集団は個別にではなく一斉に落ちる。

**CV で検証した genuine な改善は、予測可能性が高い。** 保険側の -0.0006 という小さな
差はこれを裏付けている。今回、序盤の日数をもっと自前の ensemble 強化 (feature
engineering、多様なモデルの追加、per-class threshold の調整など) に振っていれば、
土台となる genuine な実力自体を伸ばせていたはずだった。

**Bronze にすら届かなかった真因は、row-flip の選択ミスではなく、そもそもの base model
の実力不足だった。** 保険の nested CV スコアは Private でもほぼそのまま出たが、その
絶対値自体が Bronze 圏 (推定 Private 0.970 以上必要) に届いていなかった。攻めるか
守るかという戦術以前の、土台の地力の問題。

## 次に活かすこと

- Public/Private split があるコンペでは、序盤から「CV で殻を割る」ことに集中時間を
  多く割く。row-flip や vote-blend のような Public LB 依存の手法は、あくまで
  診断・学習材料としての位置づけに留める
- 公開資産を土台にする時は、その資産自体がどうやって作られたかを想像する。
  高スコアであるほど、それが Public LB へのフィードバックの結果である確率も高い
- 「同じ手法が流行っている」という事実は、その手法の有効性の証拠ではなく、
  shake-up リスクの警告として読む
