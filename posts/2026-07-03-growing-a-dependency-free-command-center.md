## TL;DR

新しい基盤に、素っ気ない `<ul>` だけの管理画面が付いていた。仕様には準拠しているが
見るに堪えない。これを、CPU/GPU/RAM の実測ゲージや電気代まで出る**ダークテーマの
運用コマンドセンター**に育てた。方針は徹底して「依存フリー」— 新規SPAもStreamlitも足さず、
既存の小さな画面モジュールをサーバーレンダHTMLのまま拡張し、外部CDNもJSフレームワークも
使わない。制約(画面はAPI経由でしか状態を読まない)を守ったまま、どこまで作れるかの記録。

---

## 選択肢は3つあった

管理画面のレンダリング方式には3つの道があった。

1. 既存の画面モジュール(サーバーレンダHTML)を育てる
2. リッチなウィジェットが速く作れる別アプリを新規に立てる
3. JS の SPA を組む

2 と 3 は見た目のリッチさでは有利だが、別プロセス・別ポート・認証の二重化・別デプロイが
ついてくる。単一のアプリとして1コマンドで起動でき、既存のコンテナにそのまま乗る、という
運用のシンプルさを壊したくなかった。だから 1 を選んだ。既存モジュールを育てる。

代償はある。リッチなグラフやライブ更新を、素のサーバーHTMLで作るのは SPA より手間だ。
でも「1つのプロセス」「外部依存ゼロ」という運用上の軽さは、その手間に見合うと判断した。

## 守った1本の制約 —「画面はAPI経由でしか状態を読まない」

この基盤には最初から1つの規約がある。**画面モジュールは、他のどのモジュールも直接
参照してはいけない。状態はすべてAPI越しに読む。** これはテストで機械的に強制されていて、
画面のソースが内部モジュールを import した瞬間にテストが落ちる。

面倒に見えるが、これが効く。画面はAPIの公開契約だけに依存するので、裏側を作り替えても
画面は壊れない。育てる間ずっと、この一本道(画面 → API → 各モジュール)を破らずに済んだ。
新しい画面が欲しければ、まずAPIに読み取り口を足し、それを画面が読む。順序が決まっていると
迷わない。

ホーム画面だけは複数モジュールの数字を1枚に集めたかったので、「集約エンドポイント」を
1つ作った。これも規約上OKで、集約した各値が結局はどれかのモジュールの公開口に辿れる限り、
ダッシュボード用に混ぜてよい、と最初から許可されている。設計の余白が仕込んであると、
こういう時に助かる。

## ハードウェア監視をどこに置くか

旧来の管理画面には CPU/GPU/RAM のリアルタイム監視があって、これは絶対に引き継ぎたかった。
問題は「telemetry をどのモジュールが持つか」。新しいモジュールを増やすのは大げさだ。

結局、外の世界に触るツール層に置いた。ここには既にGPUを扱う仕組みがあったので、CPU/RAMを
標準ライブラリで、GPUを rocm-smi / nvidia-smi のコマンド出力を解析して読むだけの
`SystemMonitor` を足した。肝心なのは**優雅に劣化する**こと。GPUの無いマシン(手元の開発機)
では例外を投げず「GPUなし」と返し、telemetry用のコマンドが無ければゼロを返す。おかげで
どのノードでも画面は描ける。開発機では「no GPU」、GPUのある実行ノードでは本物の値、という
ふうに。

APIに `/system` を足し、画面はそれを読んでゲージバー(負荷で緑→琥珀→赤に変わる)を出す。
実行ノードに載せて初めて、24GBのVRAMや温度や消費電力が本物の数字で埋まったのは気持ちよかった。

## ついでに電気代

消費電力が見えるなら、電気代も出したくなった。GPUの実測電力は取れる。CPU・基板は直接
測れないので、そこはざっくり見積もる —「ベース + CPUを使用率で idle→peak に伸ばす + GPU実測」。
これに単価(環境変数で設定、既定は日本の平均のざっくり値)を掛けて円/時にする。

これを右上に、控えめに常時出した。全ページに出したいが、各画面のハンドラに telemetry を
配線するのは違う。バッジは「ページの中身」ではなく「額縁(chrome)」だ。だから共通の枠に
プレースホルダだけ置き、~6行のインラインJSが `/system` を定期的に取ってきて数字を埋める。
これで全ページに勝手にライブで出る。ホバーするとW・円/日・単価の内訳が出る。

見積もりの精度は粗い。粗くていい。「今このノードがいくらで動いているか」を横目で掴めれば
十分だ。GPUが回ると数字が跳ねるのが分かって、そういう意味では十分に「効いている」。

## 学び

- 「依存フリー」を制約として先に決めると、判断が速くなる。ライブラリを入れれば楽な場面
  (グラフ、ライブ更新)でも、素のSVG/HTML/1つのkeyframe/meta-refresh/小さなfetchで届く
  範囲は思ったより広い。
- 「画面はAPI経由でしか読まない」という一本道は、育てる作業をずっと安全にした。裏を
  変えても画面が壊れない安心感は、遅くなる代わりのコストを上回る。
- 優雅に劣化させる設計(GPU無し→"no GPU"、コマンド無し→ゼロ)にしておくと、開発機と
  実行ノードで同じ画面がそのまま動く。環境ごとに分岐を書かなくて済む。

素っ気ない `<ul>` から、実GPUと電気代まで出る運用画面まで、外部依存を1つも足さずに
到達できた。運用画面は「軽くて壊れないこと」がリッチさより効く場面が多い、と改めて思った。
