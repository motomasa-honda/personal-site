## TL;DR

自律エージェントに「Titanicの生存予測をやって」と頼み、実際に手元のCSVを渡した。
何回か試行を繰り返した末に、それらしい精度のスコアが返ってきた。数字だけ見れば
成功に見えた。ところがコードの中身を確認すると、モデルは渡したCSVを一度も
開いていなかった。代わりにインターネットから別のデータセットを勝手にダウン
ロードして、それで学習していた。原因は「指示が何段階も言い換えられていく間に、
具体的な事実が抜け落ちる」という、LLMをパイプラインで繋いだときに起きがちな
劣化だった。直し方と、なぜ気づけたかを書く。

---

## 起きたこと

構成はシンプルだ。人間が大まかな目的を伝える → 上位のエージェントがそれを
分析方針の文章に言い換える → その方針をもとに作業計画が作られる → 計画の
各ステップの説明文をもとに、コード生成担当が実際のPythonコードを書く。

このパイプラインに「Titanicの生存予測モデルを作って、5分割交差検証の精度を
出して」という依頼を投げた。作業ディレクトリには本物の `train.csv` と
`test.csv` を置いてある。

返ってきたスコアは 0.9667。悪くない数字に見えた。が、見覚えがあった——数値
だけ拾えば、以前ダミーデータで同じパイプラインを試したときと**まったく同じ
桁まで一致していた**。これはおかしい。本物のデータで、しかも別の日に走らせて、
偶然そこまで一致することはまずない。

コードを開いて確認した。

```python
from sklearn.datasets import fetch_openml

mnist = fetch_openml('mnist_784', version=1, as_frame=False)
X, y = mnist.data, mnist.target.astype(int)
```

Titanicのコードのはずが、MNIST(手書き数字の画像データセット)を openml から
ダウンロードしていた。渡した `train.csv` はどこにも登場しない。別の試行では、
`sklearn.datasets.make_classification` でその場で人工的に作ったダミーデータを
学習させているものもあった。

一番まずいのは、この後の挙動だ。このパイプラインには「前回より良いスコアが
出たら、それを採用して次の改善に進む」という仕組みが組み込まれている。
偽物のデータで出した偽物のスコアを、本物の改善として**そのまま報酬にしていた**。
本物のデータより偽物のデータの方が高いスコアを出せば、システムは偽物の方を
「良い結果」として学習し続けてしまう。

## なぜ情報が消えたか

依頼から実際のコードが書かれるまでに、LLMの呼び出しが4段階挟まっている。

```
人間の依頼
  → (LLM) 分析方針の文章
    → (LLM) 作業計画への分解
      → (LLM) 各ステップの説明文
        → (LLM) Pythonコード生成
```

「作業ディレクトリに `train.csv` がある」という具体的な事実は、依頼文の
どこにも明示的な形では書かれていなかった——「手元にあるデータを使って」
という曖昧な言い回しだけがあった。この曖昧な言い回しは、4段階の言い換えを
経るうちに、サンプリングのたびに少しずつ形を変え、最終的には「何らかの
データセットで学習する」という抽象的な指示にまで薄まっていた。コード生成
担当からすれば、具体的なファイル名を渡されていない以上、それらしい
データセットを自分で調達するのは筋が通った振る舞いですらある。

これはLLMを直列に繋いだときに一般的に起きる劣化で、今回に限った話では
ないと思う。「言葉で説明して次のLLMに渡す」というリレーは、途中の
どこか1段階でも具体性が失われると、後段はそれを埋め合わせる術がない。

## 直し方

直感的な対策は「依頼文にファイル名をちゃんと書く」ことだが、それは
また同じリレーの中に情報を置くだけで、根本的な解決にならない。次に
別の言い回しで依頼したら、また同じように蒸発する可能性がある。

正しい直し方は、**リレーを経由させないこと**だった。コードを生成する
直前の1段階で、作業ディレクトリの中身をファイルシステムから直接読み、
そこで初めて得られる事実をプロンプトに差し込む。

```python
def data_summary(self, max_files=8):
    """作業ディレクトリの CSV を、ファイル名とヘッダ行だけ集めて返す。
    LLMの言い換えを一切経由しない、filesystem からの直接読み取り。"""
    lines = []
    for path in sorted(self._workspace.glob("*.csv"))[:max_files]:
        header = open(path).readline().strip()
        size = path.stat().st_size
        lines.append(f"- {path.name} ({size} bytes; columns: {header})")
    return "\n".join(lines)
```

これをコード生成のプロンプトに、次のような一文と一緒に必ず添える。

> 作業ディレクトリに存在するファイル(事実):
> - train.csv (61194 bytes; columns: PassengerId,Survived,Pclass,...)
> - test.csv (28629 bytes; columns: PassengerId,Pclass,...)
>
> このコードは必ずこれらのファイルを読み込んで使うこと。
> データセットのダウンロードや、人工データの生成は禁止する。

依頼文が何段階言い換えられようと、この一段落だけは毎回filesystemから
新鮮に読み直されるので、劣化のしようがない。

再実行すると、生成されたコードは

```python
train_df = pd.read_csv('train.csv')
test_df = pd.read_csv('test.csv')
X_train = train_df[['Pclass', 'Sex', 'Age', 'SibSp', 'Parch', 'Fare']]
```

ときちんと本物のファイルを読むようになり、精度も 0.81〜0.82 という、
Titanicの5分割交差検証としてありうる現実的な範囲に落ち着いた。0.9667
という「よすぎる数字」は、それ自体が異常のサインだったわけだ。

## 保険をもう一段

プロンプトを直しても、次に別のモデルや別の言い回しで同じ問題が
再発しない保証はない。なので、生成されたコードが実際にそのファイル名を
参照しているかどうかを、実行後に機械的にチェックする層も足した。

```python
data_files = ml_execution_tool.data_file_names()
if data_files and not any(name in saved_code for name in data_files):
    metadata["data_context_ignored"] = True
```

参照していなければ、そのスコアは即座に不正扱いで捨てるのではなく、
「無視した」というフラグを付けて記録に残す。スコア自体は保持する——
黙って0点にすると、何が起きたか後から追えなくなるからだ。フラグが
立った試行は、後段の評価スコアリングでも減点対象にした。データを
無視することが、単に無意味なだけでなく、明確に損になるようにした。

## 今日の教訓

数字が「それらしい」だけでは、何も証明していない。むしろ「以前見た
数字と一致しすぎている」という違和感の方が、遥かに雄弁なシグナルだった。
今回もし前回のダミーデータでの試行結果を記憶していなければ、この
バグには気づかずに「本物のデータで良い精度が出た」と誤って納得して
先に進んでいたと思う。

もう一つ、LLMを直列につなぐパイプラインでは、「具体的な事実」と
「解釈・要約・言い換え」を明確に区別しておくべきだという教訓も得た。
事実は、後段にどれだけ離れていても、リレーを経由させずに直接その場で
取得し直す。解釈だけを言葉のバトンとして渡す。この線引きを怠ると、
バトンの中に紛れ込んだ事実は、渡す人数が増えるほど確実に薄まって消える。
