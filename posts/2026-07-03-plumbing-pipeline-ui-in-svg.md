## TL;DR

自律エージェントの実行ループを可視化するのに、最初は「箱を横に4つ並べたステッパー」を
作った。動くけれど、見た瞬間に「これじゃない」と思った。ループ構造(やり直しが起きて
先頭に戻る)が平らな一列では表現できていなかったからだ。作り直して、ケーシングと流体が
流れる**本物の配管みたいなSVG図**にした。パイプに液が流れるアニメーション、reflect(振り返り)
バルブから元に戻る/抜ける分岐、複数コンペが枝分かれするマニフォールド。最終的に、実行中は
勝手にステージが進む「生きた」ダッシュボードになった。この記事はその設計の記録。

---

## 箱を並べた時点で違和感があった

エージェントは1回のイテレーションで「プランを見直す → コードを生成する → 実行する →
結果を記録する」という4ステージを回る。素直に実装すると、この4つを横に並べて、現在の
ステージを青くハイライトするステッパーになる。

作ってはみたが、平らな一列は嘘をついていた。実際のループは直線ではない。記録のあとに
**reflect** という判断が入って、「まだ続ける」なら先頭のプランに戻り、「もう止める」なら
抜ける。この「戻る」構造こそがマルチイテレーションの本質なのに、箱の一列はそれを消してしまう。

一列のステッパーが表現できるのは「今どこ」だけで、「どう回っているか」を落としていた。
これが気持ち悪かった。

## パイプは線ではなく「管」に見せる

作り直しの方針は「線ではなく管に見せる」。SVGで、1本のパイプを3層のストロークで描いた。

- いちばん外に、背景と同色の太いケーシング(縁取り)
- その内側に、少し明るいケーシング
- いちばん上に、色つきの細い「流体」

この3層を重ねるだけで、ただの線が急に「断面のある管」に見える。色は状態を表す —
通過済みは緑、実行中は青、未通過は暗いグレー。

そして流れ。稼働中のセグメントには `stroke-dasharray` で破線を乗せ、`stroke-dashoffset` を
CSSキーフレームで動かして、破線が管の中を流れていくようにした。これで「液が流れている」
感覚が出る。ライブラリは一切使っていない。SVGのパスと、キーフレーム1つだけ。

```
@keyframes flow { to { stroke-dashoffset: -34; } }
.flow { stroke-dasharray: 5 13; animation: flow .9s linear infinite; }
```

## reflect ループを literal に描く

いちばん描きたかったのがループだ。4ステージを上段に横並びにして、最後の「記録」から
パイプが下に落ち、下段を右から左へ戻り、途中の**reflect バルブ**(円形のパイプ継手)を
通って先頭のプランに上がっていく。これが「続ける」経路。バルブからもう一本、下に分岐が
落ちて「done」に抜ける。これが「止める」経路。

稼働中はループ全体の管に流れアニメが走って、配管が循環しているように見える。実行が
終わると、アニメが止まり、全ステージが緑で固まり、停止した理由(目標達成/最大回数/
同じエラーの連続)がバルブの詳細に出る。箱のステッパーでは絶対に出せなかった「どう
回って、なぜ止まったか」が、絵の形そのもので伝わるようになった。

JS のダイアグラムライブラリ(mermaid や d3)を入れれば楽に描けたはずだが、入れなかった。
トポロジは固定の4ノード + ループで、ライブラリを設定するより手でSVGを書くほうが短いし、
外部CDNへの依存もゼロで済む。運用ダッシュボードは依存フリーで動いてほしかった。

## 分岐 — マニフォールド

もう一段、「複数の系統」も欲しくなった。実行基盤は同時に複数のコンペを回せる。そこで
コンペ一覧を、1本の元管から**コンペごとに枝分かれするマニフォールド**にした。左の縦管
(元管)から、各コンペへエルボー(直角の継手)で枝が伸び、それぞれが4つのステージ・ピップ
(現在地が光る)と、スラグ・ステータスを持つ。枝はそのままそのコンペの詳細パイプラインへの
リンクになっている。

実行中の枝は流れアニメで動き、完了した枝は静止して緑になる。1画面で「今どのコンペが
どのステージにいるか」が配管の形で一望できる。

## 自走 — 実行中だけ自動リフレッシュ

最後に、手でリロードしなくてもステージが進むようにした。ここもライブラリを避け、
サーバーレンダのHTMLに `<meta http-equiv="refresh">` を、**実行中のジョブがある時だけ**
差し込んだ。完了すればリフレッシュは自分で止まる。SSEやwebsocketのようなストリーミングは
入れていない。単一オペレーターが眺める運用画面には、durableに書かれた状態を数秒ごとに
読み直すだけで十分だと判断した。

## 学び

- 可視化は「今どこ」だけでなく「どう回るか(構造)」を落とさないように描く。箱の一列が
  ループを消していたのが、作り直しの動機だった。
- 「線を管に見せる」のはストロークを3層重ねるだけ。この一手で情報量は変えずに、絵の
  説得力が段違いになった。
- 固定トポロジの図なら、ダイアグラムライブラリより素のSVG + キーフレーム1つのほうが
  短く、依存もゼロ。運用画面ではこの「軽さ」が効く。

見た目の遊びのようで、実は「エージェントが今なにをしていて、なぜ止まったか」を人間が
一目で掴めるかどうかは、運用のしやすさに直結する。配管に描き替えてから、進捗を追うのが
明らかに楽になった。
