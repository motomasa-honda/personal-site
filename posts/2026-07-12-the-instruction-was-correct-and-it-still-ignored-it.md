## TL;DR

複数の工程に分けて生成AIにコードを書かせるパイプラインで、あるファイルが
「1つ上の階層にある」とプロンプトで正しく伝えているのに、実行モデルは
何度やっても素の相対パスでそのファイルを読もうとして失敗し続けた。
最初はプロンプト内の矛盾が原因だと思って直したが、矛盾を消しても直らな
かった。結局効いたのは、指示を工夫することではなく、「AIが何を書いても
そのパスが動くようにしてしまう」という力業だった。

---

## 起きたこと

複数の工程(データ読み込み→特徴量作成→学習、のような構成)に分けて
生成AIにコードを書かせる仕組みは、以前一度直したはずだった。前工程が
実際に書いたファイル名を後工程に機械的に伝える、という修正で、ある
種類のバグは解消していた。

ところが実際に本物のモデル(30B級のコード生成モデル)へ本物の
Kaggleコンペを投げてみたところ、今度は最初の工程からして毎回同じ理由で
失敗した。

```
FileNotFoundError: train.csv
  File ".../ws/nodes/load_data/main.py", line 8, in <module>
    train_df = pd.read_csv('train.csv')
```

各工程は、コンペのデータが置いてある場所より1階層下の専用ディレクトリ
で実行される設計になっている。だから `train.csv` を読むには
`../train.csv` と書く必要がある。プロンプトにはその旨をきちんと
書いていた——はずだった。

## 1回目の修正: 実は指示が自己矛盾していた

プロンプトを組み立てているコードを読み返すと、原因はすぐに見つかった。
コード生成を呼び出す共通処理そのものが、無条件にこう言い切っていた。

```
Data files present in the current working directory (ground truth):
- train.csv (61194 bytes; columns: ...)
The code MUST load and use these files.
```

「データファイルはカレントディレクトリにある」——単一工程の実行では
正しい前提だったが、複数工程に分かれた今回は真っ赤な嘘だった。しかも
このメッセージは、別の場所で追加していた「1階層上にあります」という
ヒントと**同じプロンプトの中で**矛盾していた。AIは、より断定的で後に
出てくるこちらの主張に従っていた。

矛盾を解消し、状況に応じて言うことを一本化した。

```
Data files present at '../' (relative to the current working directory):
- train.csv (61194 bytes; columns: ...)
Reference each one as '../<filename>'.
```

これで直るはずだと思った。テストも通した。

## 直っていなかった

実機の本物のモデルにもう一度同じコンペを投げた。返ってきたコードは、
また `pd.read_csv('train.csv')` だった。矛盾のない、明確で、正しい
指示を与えたのに、モデルは無視した。

これは想定していなかった。今までの経験則は「AIに事実を教えなければ
AIは推測で埋める」というものだった。今回は事実をきちんと教えていた。
それでも埋まらなかった。おそらく、Kaggleのtitanicコンペと言えば
`pd.read_csv('train.csv')`という組み合わせが、学習データの中であまりに
強く刷り込まれていて、目の前のプロンプトの1文がそれに勝てなかったの
だと思う。

## 2回目の修正: 指示をやめて、事実の方を動かす

ここで発想を変えた。「AIに正しく従わせる」のをあきらめて、「AIが
何と書いても結果的に正しく動く」ようにした。各工程用のディレクトリが
作られた直後に、コンペのデータファイルへのシンボリックリンクを
そこへあらかじめ張っておく。

```python
def ensure_root_data_files_reachable(node_dir, root_workspace):
    node_dir.mkdir(parents=True, exist_ok=True)
    for path in sorted(root_workspace.glob("*.csv")):
        link = node_dir / path.name
        if not link.exists() and not link.is_symlink():
            link.symlink_to(path)
```

こうしておけば、AIが `train.csv` と書いても `../train.csv` と書いても、
どちらも実際に読み込める。指示に従うかどうかに結果を左右されなくなった。

実機でもう一度実行した。今度は最後まで完走し、実際の精度スコアが
出力された。

## 今日の教訓

以前、「AIに渡していない情報をAIに推測させるな、事実を機械的に注入
せよ」という教訓を得たことがある。今回起きたのは、その一段先だった。
**事実をちゃんと注入していても、AIがその事実に基づいた指示に従うとは
限らない。** 指示は「解釈してもらう」ものであって、確約ではない。

プロンプトの文言をどれだけ工夫しても、それは「お願い」でしかなく
「保証」にはならない。本当に確実に動かしたいものがあるなら、AIの
協力を前提にせず、環境そのものを「AIが何を書いても動く」形に
作り変えてしまう方が結局早い。今回はシンボリックリンク1つだったが、
考え方としては、AIへの指示を書き直す前に「これは本当に指示で
解決すべき問題か、それとも構造で解決すべき問題か」を先に自問する
べきだったと思う。
