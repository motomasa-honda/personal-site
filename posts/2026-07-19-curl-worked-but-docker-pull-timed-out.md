## TL;DR

コンテナを久しぶりに実機で起動しようとしたら、イメージのpullがTLSハンドシェイクの
タイムアウトで毎回失敗した。DNS解決、候補IPすべてへの個別疎通、プロキシ設定、
大容量ダウンロード——定番の原因を一つずつ`curl`で潰していったが、どれも「正常」と
出る。それでもDockerデーモンのpullだけが失敗し続けた。原因はまだ特定できていないが、
「何が原因でないか」をここまで絞り込めたこと自体が、次に繋がる収穫だった話。

---

## 起きたこと

開発用のLinux機で、しばらく使っていなかったコンテナを久しぶりに起動しようとした。
`docker compose up --build`を実行すると、ビルドの最初の一歩、ベースイメージの
pullでいきなり止まる。

```
failed to resolve source metadata for ghcr.io/astral-sh/uv:latest:
failed to do request: Head "https://ghcr.io/v2/astral-sh/uv/manifests/latest":
net/http: TLS handshake timeout
```

`docker.io`側のイメージに切り替えても同じ症状。手元の環境情報を頼りに、まず
「ネットワークがそもそも壊れているのでは」と疑うところから始めた。

## 疑い①: DNSがおかしいのでは

```
$ getent hosts ghcr.io
20.27.177.117   ghcr.io
```

普通に解決できている。ここは白。

## 疑い②: そのIPに実際に届いていないのでは

DNSが正しくても、実際の経路が詰まっている可能性はある。`curl`で直接そのIPに
繋いでみた。

```
$ curl -sS -4 --resolve ghcr.io:443:20.27.177.117 https://ghcr.io/
* Connected to ghcr.io (20.27.177.117) port 443
```

0.03秒程度で普通に応答が返ってくる。Dockerのpull先(CloudFront経由のCDN)が
返す複数の候補IPすべてに対しても、一つずつ同じ確認をしたが、全部同じように
一瞬で応答が返ってきた。ここも白。

## 疑い③: プロキシが変な形で挟まっているのでは

```
$ docker info | grep -i proxy
$ cat ~/.docker/config.json
cat: /home/user/.docker/config.json: No such file or directory
```

プロキシ設定はどこにも無い。systemdのDocker用ドロップインファイルも存在しない。
ここも白。

## 疑い④: 大きいデータの転送自体が詰まるのでは

pullで実際に落ちてくるのはコンテナイメージの中身、それなりのサイズのデータだ。
「小さいリクエストは通るが、大きい転送になると詰まる」という可能性を確かめる
ため、無関係な別のCDNから50MBのテストファイルを落としてみた。

```
$ curl -o /dev/null -w '%{size_download} %{speed_download} %{time_total}\n' \
    'https://speed.cloudflare.com/__down?bytes=50000000'
50000000 66140015 0.755972
```

0.75秒で50MB、余裕で通っている。大容量転送そのものは問題ではない。

## 全部白なのに、Dockerだけ落ちる

DNSも、個別IPへの疎通も、プロキシも、大容量転送も、すべて`curl`では問題なし。
それなのに`docker pull`だけが、何度試しても同じ場所でTLSハンドシェイクタイムアウト
になる。同じホスト、同じネットワーク経路のはずなのに、片方(curl)は一貫して
成功し、もう片方(dockerdのpull処理)は一貫して失敗する。

ここまで来ると、疑うべき場所が変わってくる。ネットワーク経路の問題ではなく、
Dockerデーモン自身のTLSクライアント実装(コンテナランタイムのレジストリ通信部分)
がこの環境の何かと相性が悪い、という可能性の方が高くなってくる。

## 今日の教訓

「繋がらない」という同じ症状でも、原因の切り分けは"何が正常に動いているか"を
一つずつ確認していく作業だ。DNS、個別IP疎通、プロキシ、大容量転送——これらを
順に白黒つけていったからこそ、「ネットワークは壊れていない、Dockerというアプリ
ケーション層の何かが壊れている」という、当初の予想とは違う場所まで絞り込めた。

原因がまだ完全には特定できていない(コンテナランタイムのデバッグログを直接見る、
公式配布イメージを別経路で試す、といった続きの調査が必要)。それでも、「ネット
ワーク設定をいじる」という遠回りに時間を使わずに済んだのは、疑いを一つずつ
`curl`という単純な道具で機械的に確認していったからだった。「原因が分からない」
で止まるのではなく、「ここまでは原因ではないと確認できた」を積み重ねることに
価値がある。
