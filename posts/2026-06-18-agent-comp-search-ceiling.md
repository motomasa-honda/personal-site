## TL;DR

- supervised (表データ/CV/予測 CSV) 専用に作ってきた自律 Kaggle システムが、初めて **シミュレーション(エージェント)系コンペ** に遭遇した。Pokémon TCG を「プレイするボットを提出して対戦で評価される」やつ。既存パイプラインは何ひとつ噛み合わない
- 救いはエンジンが **前方シミュレータ (search API)** を同梱していたこと。ルールを自前実装せず探索が組める。これは決定的だった
- ボットを random → greedy → hybrid → **determinized Monte Carlo** と積み上げ、実コンペに提出して **LB 中央値超え** までは行った
- そこから「次の一手」を探って **計算量・デッキ・サブ決定ヒューリスティック** の3つを試したら、**全部が頭打ち or 中立**。単純探索+ルールの天井は中央値付近だと実測で分かった
- 一番おもしろかったのは determinization の検証。flat MC は最初 hybrid に **負けて (45%)** いたが、隠れ情報のサンプリングを直したら **62% で逆転**。しかも **相手モデルをわざと誤らせても 62% を維持** = 強さは「相手を読めていること」ではなく探索の構造そのものだった

## supervised システムがエージェントコンペに出会う

うちの Kaggle 自律システムは、これまでずっと「train/test をダウンロードして、CV を回して、予測を `submission.csv` に吐く」表データ系のために作ってきた。アンサンブルも full-data 再学習も、全部この前提の上に乗っている。

ところがあるシミュレーションコンペを覗いたら、データに train も test も target も無い。代わりに入っていたのは **ゲームエンジン本体**と、`agent(observation) -> 選択肢のindex` という関数を実装した `main.py` だった。提出するのは予測 CSV ではなく、**ゲームをプレイするボット一式**。評価は他人のボットとの対戦。完全に別パラダイムで、既存の analyze / cv_simulate / ensemble / submit が全部空回りする。

普通ならここで「対象外」と切り捨てる。でも一つだけ追い風があった。エンジンが **前方シミュレータ** を API として提供していたのだ。

```text
search_begin(観測, 自分の山/手札/サイドの予測, 相手の予測) -> 探索状態
search_step(探索状態, 選択) -> 次の探索状態
```

これは効く。カードゲーム AI の最大の壁は「数千種のカード効果をルールとして実装すること」なのに、それをエンジンが肩代わりしてくれる。**ルールを書かずに先読みできる**なら、探索ベースのボットが現実的に組める。

## ボットのラダーを積む

まず対戦ハーネス (deck × deck → 勝者) を薄く包んで、**自己対戦の勝率を「CV の代わり」** にした。1戦が 10ms 未満で終わるので、数千戦の勝率評価が一瞬で回る。これでボットの強さを定量比較できる足場ができた。

そこから順に積む。

**random** はベースライン。**greedy** は素朴に「最大ダメージの技で殴る」だが、最初に作った版は random に **1%** しか勝てなかった。敗因をログで追うと、全敗が「アクティブにポケモン不在」での負け。greedy が攻撃を最優先するせいで、毎ターン即殴ってターンを終え、**ベンチを一切育てない**。唯一のアタッカーが気絶した瞬間に控えがいなくて負けていた。

攻撃はターン終了を伴う。だから「展開(プレイ/エネ付与/進化)を先に全部やってから、最後に攻撃する」順序に直す。これだけで **94%** に跳ねた。

次の **hybrid** は「展開手順は greedy のまま、攻撃の的だけ先読みで選ぶ」。静的なダメージ表ではなく、技を撃った結果の盤面を1手シミュレートして評価する。弱点 2 倍や効果込みの実ダメージで技を選べるので、vs random 98% / vs greedy 56.5%。

## determinization で殴られて学んだこと

本命は **Monte Carlo**。各候補手について、エンジンの探索 API で終局までプレイアウトを回し、勝率が最大の手を選ぶ。

ところが最初の flat MC は **hybrid に 45% で負けた**。先読みしているのに、しないボットより弱い。

原因は **determinization** だった。探索 API は隠れ情報(相手の山・手札・サイド)の予測入力を要求する。最初の実装はそこを「基本ポケモン1種で雑に埋める」だけで、しかも候補手ごとにバラバラにサンプリングしていた。

直したのはこの2点:

- **隠れ情報をちゃんとサンプルする**: 自分の山は「既知の60枚 − 見えているカード」から、相手は mirror prior(=自分のデッキ構成)から引く
- **同一サンプル下で全候補手を比較する** (ペア化して分散を減らす) + サンプルごとに引き直す

結果、vs hybrid が **45% → 62%** (120戦、+2.6σ で有意)。

ここで気になったのが「この 62% は自己対戦(相手=自分と同じデッキ)だから出ているだけで、別デッキの実コンペでは消えるのでは?」という疑い。そこで **相手モデルをわざと誤らせて** (mirror ではなく雑な filler prior にして) 測り直した。

```text
MC(正しい相手モデル) vs hybrid : 63.3%
MC(誤った相手モデル) vs hybrid : 61.7%   ← ほぼ変わらない
```

優位は **相手を読めていることから来ていなかった**。ペア化+リサンプルという**探索の構造そのものの実力**だった。最初の flat MC が 45% だったのも「prior が雑」ではなく「構造が未熟」だったから、と切り分けられた。こういう「効いている理由」を反証込みで詰められると、次の判断が一気に楽になる。

## 実コンペに出して現実を見る

ボットを自己完結の `main.py` に組み込み、tar.gz で実提出した。提出形式は不明だったが、`main.py + エンジン + デッキ` をアーカイブのルートに置く形で通った。

学んだ点:

- **LB は動的レーティング(Elo 的)**。提出直後は暫定値で、対戦が積まれると収束・変動する。最初 600 だったボットが、しばらくして 712 まで上がった
- 自己対戦で強いボット(MC)を出したら、実 LB でも hybrid を上回った。**「自己対戦の勝率 → 実 LB」の転移が方向として確認できた**。これは表データの CV-LB 相関と同じ話で、代理指標が本番に効くと分かれば、以降の最適化を自己対戦で回してよい根拠になる
- 提出前に「スコアが上がる目処」を立ててから出す規律も効いた(上の determinization 反証はその一部)

到達点は **LB 中央値超え**。初参戦のボットとしては悪くない。が、トップはそのざっと2倍のレーティングで、まだ遠い。

## 「次の一手」を3つ試して、3つとも頭打ちだった

中央値の先へ行くために、思いつくレバーを順に引いた。全部、自己対戦で定量検証した。

| レバー | 仮説 | 結果 |
|---|---|---|
| 探索の計算量を倍 | サンプルを増やせば賢くなる | **頭打ち** (n_det 16 vs 6 = 40%、増やしても強くならない) |
| デッキ構築 | サンプルデッキはエネ過多(60枚中35枚)で弱い | **中立** (エネ削減+サーチ追加でも MC 50%) |
| 賢いサブ決定 | discard は低価値を捨て、search は重要札を取るべき | **中立** (smart vs dumb = 46.5%) |

3つ目が地味に効かなかった理由がおもしろい。Ultra Ball の「手札2枚を捨てる」コストで重要札を捨てない、サーチでポケモンを取る、といった「賢い選択」を実装して A/B したら差が出なかった。観察すると **エンジンが選択肢を最初からそこそこ良い順で並べていた**。だから「先頭を取る」素朴な実装が、価値ベースの選択とほぼ同じ手になっていた。

3連続の負け(or 中立)で結論は明確になった。**「単純なルール + flat Monte Carlo」という枠組みは、中央値付近で天井に当たる。** greedy プレイアウトで終局の勝敗を測る、という評価信号そのものに上限がある。サンプルを増やしても、デッキを変えても、選択を丁寧にしても、同じ天井を別の角度から触っているだけだった。

## 天井の破り方

ここから先は質的に別物が要る。本命は **学習した評価関数** を探索に組み込むこと(AlphaZero の value だけ版)。

- 自己対戦で大量の (局面, 最終結果) を作る(対戦ハーネスは既にある)
- 局面を固定長ベクトルに特徴量化して、価値ネット(勝率予測)を学習する
- Monte Carlo の「greedy で終局まで回す」leaf 評価を、この**学習価値**に置き換える。回さずに評価できるので速くて、上限も上がる

設計上の地味な勘所として、**提出環境に学習フレームワークがある保証はない**。なので学習は GPU で回しても、**重みは numpy に書き出して、提出する `main.py` の中では手書きの forward(行列積+ReLU+sigmoid)で推論**する。小さい MLP なら数行で済むし、依存も増えない。

合否ラインは「学習価値ボットが現チャンピオン(MC)を自己対戦で超える」こと。超えたら、また目処を立ててから実 LB に出す。負の結果を3つ踏んで、ようやく「ここを殴るべき」という壁の場所が分かった、という回だった。
