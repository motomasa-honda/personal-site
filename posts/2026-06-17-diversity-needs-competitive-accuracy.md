## TL;DR

- 前回「GBDT ばかりの自動探索に異系統 (線形) を混ぜれば ensemble が効くはず」と書いた。今回その仮説を実機で検証した
- 最初の異系統 trial は CV 0.1486 と GBDT (0.1287) に大きく負けた。hill climbing は**この弱いモデルに重みを一切与えなかった** — 貪欲最適化として正しい判断
- **低相関でも、単体精度が大きく劣るモデルはブレンドに入らない**。多様性は「精度が拮抗して初めて」効く
- 線形が弱かったのは前処理が稚拙だったから。`RidgeCV` + 歪み特徴の log 変換 + 全カテゴリ one-hot に直したら CV 0.1330 まで上がり、**hill climbing が 1/3 の重みを割り当てた**
- ついでに `LassoCV` が高次元 one-hot で収束せず 1 候補 25 分暴走する事故を踏み、探索の per-candidate timeout も短縮した

## 仮説: 異系統を混ぜれば効く

前回、ensemble が伸び悩む真因を「探索方向が全部 GBDT で中身が似すぎている」と推定し、`branch_explorer` の探索方向に「異系統モデル」(回帰なら Ridge / Lasso / ElasticNet) を足した。今回はその end-to-end 検証だ。

理屈の上では、GBDT と線形は**間違え方が違う**。木は階段状に、線形は滑らかに予測するので、誤差が打ち消し合ってブレンドが効く — はずだった。

## 現実: 弱い線形は一票も入らなかった

最初の検証で異系統 trial (Ridge) は確かに生成・実行された。が、CV はこうなった。

```
candidate (異系統): cv=0.14864   # ← GBDT の 0.1287 に大きく負け
[hill_climb] step 0: +trial_1   score=0.12932
[hill_climb] step 1: +trial_121 score=0.12565
[hill_climb] step 2: +trial_1   score=0.12515
[hill_climb] weights = {1: 0.5, 110: 0.25, 111: 0.25}   # ← 異系統 trial の重みはゼロ
```

hill climbing (Caruana の貪欲法) は「ブレンドに加えると CV が下がる」モデルだけを採用する。CV 0.1486 の Ridge は、いくら GBDT と低相関でも、**加えると平均が悪化する**ので一度も選ばれなかった。

これは最適化として完全に正しい。低相関は誤差を打ち消す前提条件にすぎず、**単体精度が離れすぎていると、打ち消しより足を引っ張る方が勝つ**。「多様性が大事」は半分しか正しくなくて、正確には**「精度が拮抗した上での多様性が大事」**だった。

## 線形が弱かったのは前処理のせいだった

ここで諦めず、なぜ Ridge が 0.1486 なのかを見た。生成コードの前処理が雑だったのだ。

- 固定 `alpha=10` の決め打ち (探索なし)
- `PolynomialFeatures(interaction_only=True)` を全数値列に展開して次元爆発
- 歪んだ価格系の特徴量を log 変換していない

House Prices の上位解法は、Lasso / ElasticNet を**ちゃんと前処理して** GBDT と拮抗させ、それをスタックする。つまり線形が弱いのではなく、線形に必要な下ごしらえをしていなかっただけだった。そこで異系統方向のプロンプトを、競争力が出る前処理を要求する文言に書き換えた。

```python
"異系統モデル — GBDT と精度が拮抗する低相関メンバーを作る。弱い線形は "
"hill_climb の貪欲ブレンドに採用されず無意味なので、前処理を尽くして GBDT 並みの精度を出す。"
"回帰は RidgeCV(alphas=[0.1,1.0,10.0,30.0,100.0]) を第一選択 (閉形式で速く収束問題が無い)。"
"... カテゴリ列は get_dummies で one-hot 化し train/test の列を揃える。"
"歪んだ数値列 (skew>0.75) は np.log1p、ターゲットも log1p。"
"interaction-only PolynomialFeatures の全列展開は次元爆発で精度を落とすので使わない。"
```

ポイントは `RidgeCV` を第一選択にしたこと。alpha を内部 CV で自動選択でき、かつ閉形式で解けるので速くて収束問題がない。

## 結果: 線形がブレンドに入った

書き換え後の検証で、異系統 trial はこうなった。

```
candidate (異系統 RidgeCV): cv=0.13298   # ← 0.1486 から大幅改善
[hill_climb] step 0: +trial_1   score=0.12932
[hill_climb] step 1: +trial_111 score=0.12472   # ← trial_111 = 線形。採用された
[hill_climb] step 2: +trial_110 score=0.12462
[hill_climb] weights = {1: 0.333, 111: 0.333, 110: 0.333}   # ← GBDT + 線形 + FE-GBDT
```

CV 0.1330 はまだ GBDT (0.1287) には届かない。**それでも hill climbing は線形 trial に 1/3 の重みを与えた**。単体では負けているが、GBDT と十分に低相関なので、ブレンドに加えると CV が 0.12472 → 0.12462 と下がったからだ。これが「精度が拮抗して初めて効く多様性」の実物だった。GBDT + 線形の正真正銘のブレンド ensemble が、CV 0.12462 で出来上がった。

(この GBDT + 線形 ensemble の LB 確定は、その日の Kaggle 提出枠を使い切ってしまい翌日に持ち越しになった。が、「弱すぎると入らない / 拮抗すれば入る」というメカニズムは両方向から確認できた。)

## おまけ: LassoCV が 25 分暴走した

最初に `LassoCV` / `ElasticNetCV` を許したら、フルの one-hot (数百次元) で座標降下が収束せず、**1 つの候補が 25 分走ってプロセス強制終了**された。探索全体がそこで止まる。

```
[runner] exit=-9 cv=None duration=1500.1s   # ← SIGKILL。per-candidate timeout が 1500 秒だった
```

対策を 2 つ入れた。

- 線形は閉形式の `RidgeCV` を第一選択にし、座標降下系 (Lasso/ElasticNet) を使う場合も `n_alphas` と `max_iter` の上限を必須にした
- 探索の per-candidate timeout を **1500 秒 → 300 秒**に短縮。健全な単一モデルは数秒〜数十秒で終わるので、収束しない候補を 5 分で打ち切れば探索全体のストールを防げる

## 学び

- **多様性は精度が拮抗して初めて効く**。低相関は必要条件であって十分条件ではない。単体精度が離れすぎたモデルは、貪欲ブレンドに正しく無視される
- 貪欲な ensemble は「加えると悪化するものは入れない」ので、**メンバーが弱いことを weights=0 で教えてくれる**。重みを見れば多様性が機能したか一目で分かる
- 「線形は弱い」ではなく「**線形に必要な前処理をしていない**」だった。木が許す手抜き (NaN 放置・スケール無視・カテゴリそのまま) を線形に持ち込むと自滅する
- 自動生成に重い CV 付き推定器 (`LassoCV` 等) を許すなら、**収束しない暴走を時間で打ち切るガードを必ず用意する**。閉形式で済むなら閉形式 (`RidgeCV`) を優先する
